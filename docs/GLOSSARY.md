Every FreeLayer term: what it means simply, what it means technically, and whether it exists yet.

| Term | Simple explanation | Technical explanation | Status |
| --- | --- | --- | --- |
| **Capsule** | A sealed digital envelope carrying one update between devices | Versioned encrypted envelope with algorithm identifiers, opaque routing hint, padded payload — the only cross-device format ([ADR-0003](adr/ADR-0003-capsules-as-only-cross-device-format.md), [CAPSULENET.md](CAPSULENET.md)) | Type placeholders only; wire format blocked by Gate E |
| **Blind Courier** | Any delivery channel that carries the sealed envelope without reading it | Untrusted `Transport` implementations (relay, file, QR, USB, LAN, future Tor/radio) carrying opaque ciphertext with per-transport leakage labels ([NETWORK_MODEL.md](NETWORK_MODEL.md)) | Interfaces + test-only mock; no real transport |
| **Sovereign Room** | A private workspace that lives on members' devices, not on a company server | Local-first, event-sourced room holding typed objects (messages, docs, tasks, decisions…), synced via capsules, no homeserver ([SOVEREIGN_ROOMS.md](SOVEREIGN_ROOMS.md), ADR-0006) | Type scaffolding; implementation blocked by Gate H |
| **RoomOS** | The idea that a room is a small private operating space, not just a chat | The room object model + operation log + policy that turns conversations into operational spaces | Local foundation implemented (TECH-16…20): events + replay, object model, privacy-safe queries, and unverified membership + non-authoritative capability scaffolding — all local, memory-only, policy-gated; no sync/crypto/identity |
| **Room membership (placeholder)** | A local note that someone is in a room — not a verified account | Local, UNVERIFIED `RoomMembershipRecordV1` with a placeholder role; authorizes nothing by itself (TECH-20) | Local scaffolding; verified identity is Gate G |
| **Capability descriptor** | A description of what a member may attempt — not a key or token | Non-authoritative `RoomLocalCapabilityDescriptorV1`, bound to room+membership+revision, attenuable, never serialized/persisted; only a `PolicyDecision` authorizes (TECH-20) | Scaffolding; a real capability runtime is Gate B |
| **Local revocation** | Turning off someone's local access when their membership/role/policy changes — on this device only | Membership/policy/mode changes bump a local revision fence; `assertPreparedRoomAuthorizationCurrentV1` revalidates before every side effect (TECH-21) | Local only; **not** distributed revocation (Gate H) |
| **Prepared authorization** | A checked-but-not-yet-executed permission that must be re-checked at the moment of use | Non-authoritative, transient `PreparedRoomAuthorizationV1` bound to the operation + a revision fence; never a token/cache (TECH-21) | Local; single-use/nonce-bound decisions are Gate B |
| **DevicePosture** | How much a device's security can be trusted — supplied later by the separate Secure Device project | `unverified`/`basic`/`hardened`/`high_assurance`/`managed_bunker`/`at_risk`; core resolves only `unverified`/`at_risk` (no provider); an environment attribute, never identity (TECH-22) | Contract only; verification is the Secure Device Integration Gate |
| **Room governance** | Owners tightening a room's rules — never loosening them | Tighten-only `RoomPolicyDocumentV1` changes via `applyLocalRoomGovernanceUpdateV1`; owner-placeholder is unverified; monotonic + policy-revision-bound (TECH-22) | Local; no voting/consensus/signed/distributed governance (Gates F/G/H) |
| **Policy Engine** | The rule system that decides whether the app may store, send, preview, copy or show something | Core pipeline (validate → classify → resolve → strictest wins → `PolicyDecision` → execute → audit); side-effect modules reject calls without a decision ([ADR-0002](adr/ADR-0002-core-enforced-policy-engine.md)) | Types + fail-closed placeholder; real engine is Gate B |
| **Privacy Mode** | A protection level for your whole device: Standard, Private, Ghost, Bunker, Offline Capsule, Emergency, Sovereign Room | Policy profiles over 13 capabilities, resolved strictest-wins ([PRIVACY_MODEL.md](PRIVACY_MODEL.md)) | Schema typed; enforcement is Gate B |
| **Identity Firewall** | No phone number, email or central account required — ever | Local/ephemeral identities, per-contact aliases, one-time invites, QR verification, recovery kit; no server-side user database | Design only (Gate G) |
| **Metadata Firewall** | Fewer silent leaks: typing, read receipts, online status, link previews | Policy-gated application signals + envelope/padding discipline + per-transport leakage labels ([METADATA_MODEL.md](METADATA_MODEL.md)) | Design + 10 defined invariants; tests come with transports |
| **Storage Firewall** | The app can't quietly save things it shouldn't | The write barrier: all persistence flows through StoragePolicy with a `PolicyDecision`; memory-only and null backends exist ([STORAGE_MODEL.md](STORAGE_MODEL.md), ADR-0005) | Interfaces + memory/null providers with runtime guards |
| **Network Firewall** | The app can't quietly talk to the internet | The network side-effect barrier: no connection without NetworkPolicy + mode approval + PBOM entry ([NETWORK_MODEL.md](NETWORK_MODEL.md)) | Design; Gate D |
| **ScreenShield** | Future protection for screenshots, screen recording, clipboard and risky devices | Endpoint Defense Layer: protection levels (off→bunker), capture-aware redaction, platform secure surfaces with honest capability reporting ([SCREENSHIELD.md](SCREENSHIELD.md), ADR-0012) | Research/design only |
| **ProtectedContent** | The only door sensitive content is allowed to appear through | Policy-checked rendering surface (`contentRef` + `PolicyDecision`); raw rendering of sensitive fields is a defect ([PROTECTED_CONTENT_POLICY.md](PROTECTED_CONTENT_POLICY.md)) | Contract designed; not implemented |
| **Ghost Vault** | Future mode keeping your identity keys on an offline device | Cold-identity architecture: the online client couriers encrypted capsules without holding private identity keys | Future design; architecture reserves space |
| **Local AI** | Optional future AI that runs only on your device and is off by default | Policy-gated local inference (summaries, task extraction, Q&A) behind AIPolicy + AI Privacy Guard; zero network calls ([LOCAL_AI.md](LOCAL_AI.md), ADR-0007) | Interfaces + always-rejecting provider |
| **PBOM** | A public list of everything the app actually does with your data | Privacy Bill of Materials: endpoints, storage, permissions, caches, logs, AI behavior — anything missing from it is a bug ([PBOM.md](PBOM.md)) | Live document; automation planned |
| **Threat Model** | The honest list of who could attack you and what FreeLayer can/can't do about it | Assets, attacker classes, in/out of scope, per-component risks ([THREAT_MODEL.md](THREAT_MODEL.md)) | Live document |
| **Trust Center** | The one page that answers "can I trust this today?" (currently: no) | Verified maturity/status per area, updated only when verified reality changes ([TRUST_CENTER.md](TRUST_CENTER.md)) | Live document |

## Identity terminology (RESEARCH-ID-01, research)

FreeLayer keeps **person**, **local identity / identity root**, **profile**, **alias** (per-contact / per-room), **member reference** (`RoomMemberRef`, a local unverified placeholder — *not* identity), **device** / **device key** / **device passport** (future), **DevicePosture** (external environment attribute — *not* identity), **contact**, **trust record / Trust Notebook**, **verification** (proof of a specific claim such as key control — *not* personhood), **recovery** (may reduce assurance; never an admin/master key), **authentication**, and **real-world identity proofing** (*not implemented by default*) as **distinct** terms. Full definitions: [research/IDENTITY_TERMINOLOGY_MODEL.md](research/IDENTITY_TERMINOLOGY_MODEL.md). Nothing here is implemented — identity is Gate G, decided by TECH-ID-02.

## Identity architecture terms (TECH-ID-02 / ADR-0013)

**Local identity root** (private local authority — never a public identifier) · **Persona** (presentation/policy context under one root — **not guaranteed unlinkable**) · **Independent identity root** (separate root; the real separation boundary) · **Pairwise relationship** (per-contact relationship + peer-facing pairwise identifier) · **Room identity binding** (root/persona to room to room alias to membership) · **Device authorization** (subordinate, revocable device grant) · **Device passport** (future signed root-to-device authorization; not attestation, not implemented) · **Trust Notebook** (local, non-authoritative observations) · **Recovery kit** (offline; no admin/master key; reduces assurance). Architecture decided by [ADR-0013](adr/ADR-0013-identity-firewall-architecture.md); **not implemented**. `RoomMemberRef` is not identity; **DevicePosture is not identity**.

## Identity scaffolding (TECH-ID-03)

**`@freelayer/identity`** — local, non-cryptographic identity scaffolding. **LocalIdentityRoot** (private, memory/null, no keys), **Persona** (local context, not unlinkable), **PairwiseRelationship** / **RoomIdentityBinding** (placeholders), **IdentityLocalRevision** (positive local counter, not a clock/version/vector-clock). No crypto, no persistence, no aliases/recovery/invites yet. DevicePosture is not identity; a RoomMembershipId is not identity proof.

## Ephemeral identity (TECH-ID-04)

**Ephemeral identity** — an INDEPENDENT current-process identity root (`rootKind:"ephemeral_current_process"`), bounded local lifetime, process-epoch bound, no recovery/promotion/export/sync/persistence, destroyed fail-closed. **Process epoch** — an injected, non-persisted, non-peer-facing current-process marker (mismatch = expired). Not a persona, not an alias, not anonymity, not forensic erasure, not remote deletion. DevicePosture is not identity; not safe for real secrets.

## Per-contact aliases (TECH-ID-05)

**Pairwise presentation alias** (`PairwisePresentationAliasV1`) — a LOCAL, relationship-scoped, non-cryptographic display name for how the local user presents within ONE pairwise relationship. In TECH-ID-05 it is local-only and never shared (`sharingState:"not_shared_tech_id_05"`), not authenticated (`authenticatedBinding:"not_implemented_gate_f"`), and not remotely updatable; lifecycle `draft_local → active_local_unshared → retired_tombstone`, ≤1 active per relationship, rotation preserves the relationship id / block / trust. It is **not** identity, verification, authentication, a public username, or a cryptographic identifier, and does not prove key control.

**Local peer label** (`LocalPeerLabelV1`) — a PRIVATE note the local user attaches to a contact (`visibility:"local_only"`, `peerShared:false`, no authority, no verification evidence); lifecycle `active_local_private → cleared_tombstone` (no retained text), ≤1 active per relationship. **A local peer label is NEVER sent to the peer.**

These are the local, metadata-only **per-contact alias** foundations only (the broader per-contact-alias feature in the Identity Firewall table above stays design-only): normalization is Unicode NFC + trim with dangerous control / bidi / zero-width / NUL rejection (retains ZWJ/ZWNJ; not UTS #39, no confusable detection — not proof an alias is safe); cross-relationship reuse raises a local privacy warning only (never denies/merges/reveals; not proof of linkage or of unlinkability). Remote sharing, public directory, global username, authentication, cryptographic binding, notifications, telemetry, and AI are structurally unavailable. Deferred: crypto (Gate F), wire/exchange (Gate E), full Identity Firewall (Gate G), sync (Gate H); Secure Device / ScreenShield / anti-spyware is externalized. Room membership is not identity proof; DevicePosture is not identity; not safe for real secrets.

## Per-room aliases (TECH-ID-06)

**Room presentation alias** (`RoomPresentationAliasV1`) — a LOCAL, room-scoped, non-cryptographic display name: the local presentation chosen for ONE identity room binding in ONE room. In TECH-ID-06 it is local-only and never shared (`sharingState:"not_shared_tech_id_06"`), not authenticated (`authenticatedBinding:"not_implemented_gate_f"`), and not remotely updatable; lifecycle `draft_local → active_local_unshared → retired_tombstone`, ≤1 active per binding, rotation preserves the room identity binding / membership / role / assurance (no remote deletion, no authenticated update, no historical-message relabelling). It is **distinct from identity, membership, and role** — not identity, verification, authentication, a `RoomMemberRef`, or a global username, and does not prove key control; a name collision is not proof of impersonation.

**Room identity binding** (`RoomIdentityBindingV1`) — the link from a root or persona to ONE room; **distinct from identity, membership, and role** (it is not a role and not personhood proof). Its `roomAliasState` moves off the `"not_implemented_tech_id_06"` placeholder to `RoomAliasBindingStateV1` (`"none" | "local_alias_v1" | "retired"`); the alias record is authoritative and never directly mutates RoomOS membership.

**Room-local disambiguation** — the room-scoped duplicate-name handling that forces a safe presentation when two members would render alike; **distinct from identity, membership, and role**, and distinct from verification — a collision is not proof of impersonation and disambiguation is not verification. It never resolves to `alias_only` on a duplicate and never exposes a root / global / phone / email / raw-membership id; cross-room reuse is a local redacted warning only (no room ids or counts).

**Room member display context** — the role / state / assurance a member carries in a room, held as SEPARATE fields from the alias text; **distinct from identity, membership, and role** in that the presented name is never derived from it and trust/verification is never derived from the name. A room role is not proof of identity.

These are the local, metadata-only **per-room alias** foundations only (the broader per-room-alias feature in the Identity Firewall table above stays design-only): normalization reuses the TECH-ID-05 Unicode pipeline unchanged (NFC + trim; dangerous control / bidi / zero-width / NUL rejection; retains ZWJ/ZWNJ; not UTS #39, no confusable detection — baseline Unicode validation is not full spoofing prevention); there is no automatic reuse from a contact alias, persona label, another room, a root id, or a `RoomMemberRef`. Remote sharing, authenticated binding, public directory, global username, persistence, notifications, telemetry, and AI are structurally unavailable (6 `identity.room_alias.*` scopes; `check:no-room-alias-bypass`; +12 Policy Matrix rows → 253 specs / 1771 rules; 37 new tests, 711 total). Deferred: wire/exchange (Gate E), crypto (Gate F), full Identity Firewall (Gate G), sync (Gate H); Secure Device / ScreenShield / anti-spyware is externalized. Room membership is not identity proof; DevicePosture is not identity; not safe for real secrets.
