# TECH-ID-07 — Device Key Model Threat Model

> **Sourcing / scope caveat (read first).** This is a **documentation-only, pre-implementation** threat model for the TECH-ID-07 **Device Key Model** (`packages/identity/src/devices/`, introduced by this task per [TECH_ID_07_PRECHECK.md](TECH_ID_07_PRECHECK.md) §3–§4). It adds **no cryptography**: no key bytes, no signatures, no cryptographic device authorization, no remote/cryptographic revocation, no hardware binding, no attestation, and no endpoint guarantee. Real key generation and any signed root↔device authorization are **Gate F**; the **Device Passport** is **TECH-ID-08**; synchronization / distributed revocation is **Gate H**; wire/remote exchange is **Gate E**; endpoint protection is **external** (Secure Device / Endpoint Defense, [ADR-0012](../adr/ADR-0012-endpoint-defense-layer.md)). Structural decisions: [ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §5–§7 (architecture / identifier model), §11 (multi-device model). Companion models: [TECH_ID_06_PER_ROOM_ALIAS_THREAT_MODEL.md](TECH_ID_06_PER_ROOM_ALIAS_THREAT_MODEL.md) (format reused here), [TECH_ID_04_EPHEMERAL_IDENTITY_THREAT_MODEL.md](TECH_ID_04_EPHEMERAL_IDENTITY_THREAT_MODEL.md) (independent roots, memory/null retention, process-epoch binding), and [RESEARCH_ID_01_IDENTITY_THREAT_MODEL.md](RESEARCH_ID_01_IDENTITY_THREAT_MODEL.md) (broad identity threat catalogue). **No complete mitigation is claimed for any threat below.**

> [!IMPORTANT]
> Read the **[Explicitly NOT provided](#explicitly-not-provided-read-this)** section first. TECH-ID-07 covers one **local, non-cryptographic, metadata-only** artifact and nothing more:
>
> - A **Device Authorization** is *a local record asserting that exactly one Local Identity Root subordinately authorizes one device context* ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §11). In TECH-ID-07 it holds **no key material**, is **not** cryptographically authorized, is **not** transmitted or synchronized, and grants **no** remote effect. The `DeviceAuthorizationRef` brand already exists as a declared future reference only — "never key material" ([`identifiers.ts`](../../packages/identity/src/identifiers.ts)).
>
> **A DeviceKey is not DevicePosture; DevicePosture is not identity; a device is not a person; a device reference is not a hardware identifier; a device label is not authority; a local authorization record is not a cryptographic proof; RoomOS membership is not device authorization; device revocation is not remote erasure.** Real keys / signatures / device-authorization proofs are **Gate F**; the Device Passport is **TECH-ID-08**; cross-device sync and distributed revocation are **Gate H**; endpoint custody is **external** (Secure Device).

## How to read this model

Each category lists a concrete **attack**, then **why it fails / residual** (the structural, policy, or gate reason the attack is blunted, and what it leaves behind), and the **control** that carries the weight. A control is one of:

| Control kind | Meaning |
| --- | --- |
| **Design** | A structural invariant / type boundary / reducer discipline the TECH-ID-07 devices module (`packages/identity/src/devices/`) must uphold in `@freelayer/identity` — a Device Authorization that references **exactly one** root and is **subordinate**; **opaque non-hardware** device refs (reusing the opaque-id discipline of [`identifiers.ts`](../../packages/identity/src/identifiers.ts)); **one slot = one purpose** with purpose immutable after activation and `keyReuseAcrossPurposes: "forbidden"`; an `algorithm: "not_selected_gate_f"` placeholder; a **single bootstrap** device record that is `local_unverified` / `cryptographicallyAuthorized: false` and cannot be repeated; **restrict narrows** scope and **bumps revision** ([`IdentityLocalRevision`](../../packages/identity/src/identifiers.ts)); **revoke is a terminal tombstone**; **memory/null** retention; **redacted** summaries; **content-free** errors. |
| **Policy** | A fail-closed decision resolved against an authentic exact-scope `PolicyDecision` — the eight `identity.device.*` scopes TECH-ID-07 introduces (precheck §3) — with device **add / link / passport** operations held **future-gated** in the Policy Matrix (`identity.device_key_future` → op `identity.device_key`; `identity.device_passport_future` → op `identity.device_passport`; both `effect: "future_gate"`, [`policyMatrix.ts`](../../packages/privacy/src/policyMatrix.ts)). DevicePosture may only **tighten**, never grant ([`device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts)); unknown purpose/scope **denies**. |
| **Gate** | Not solvable by this module: **Gate F** (crypto / key bytes / signatures / device-authorization proof / verification fingerprints — [ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §19), **TECH-ID-08** (Device Passport — future signed root↔device authorization), **Gate H** (sync / distributed device revocation — ADR §19), **Gate E** (wire / remote exchange), **External** (Secure Device / endpoint), **Inherent** (disclosed trade-off). |

**Code anchors used repeatedly.** Opaque refs + revision — [`identifiers.ts`](../../packages/identity/src/identifiers.ts) (opaque local ids encode **no** email/phone/username/display name, **no** key material, **no** device model, **no** timestamp-as-claim; grant **no** authority, prove **no** identity; `DeviceAuthorizationRef` is a future-reference brand, **never key material**; `IdentityLocalRevision` is a positive local optimistic-concurrency counter — **not** a timestamp, vector clock, tamper resistance, synchronization, or cryptographic continuity; `nextIdentityRevision` / `FIRST_IDENTITY_REVISION` reused for device-authorization revisions). Boundary validator — [`identity-validation.ts`](../../packages/identity/src/identity-validation.ts) (`FORBIDDEN_FIELDS` rejects `hardwareId`, `deviceFingerprint`, `devicePosture`, `deviceSafe`, `attestation`, `publicKey`, `privateKey`, `secretKey`, `seed`, `isOwner`, `capabilities`, …; `assertExactKeys` refuses mass-assignment; content-free `IdentityValidationError`). Root record placeholders — [`identity-types.ts`](../../packages/identity/src/identity-types.ts) (`LocalIdentityRootV1.deviceAuthorizationState = "not_implemented_gate_f_g"`, `keyMaterialState = "not_implemented_gate_f"`; `LocalIdentityRootKindV1 = "long_lived_local" | "ephemeral_current_process"`; root lifecycle includes `compromised_suspected` / `revoked_tombstone`; `IdentityAssuranceStateV1` includes `device_change_warning_future`; `PRODUCTION_ASSURANCE_STATES` limited to `unverified_local` / `compromised_suspected` / `revoked`). Ephemeral parity — [`ephemeral/ephemeral-types.ts`](../../packages/identity/src/ephemeral/ephemeral-types.ts) (`deviceAuthorizationState: "not_implemented_gate_f_g"`; process-epoch `processBinding`; `persistence: "forbidden"`). RoomOS membership boundary — [`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts) (`RoomMembershipRecordV1` carries **NO** device key / public key / fingerprint; `removed_tombstone` terminal via `assertMembershipTransition`; `verification: "unverified_placeholder"`) and role eligibility "**necessary but never sufficient**" ([`membership/membership-roles.ts`](../../packages/rooms/src/membership/membership-roles.ts)). DevicePosture — [`policy-composition/device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts) (`mayOnlyTighten: true`, `trustedForElevation: false`, `providerIntegrated: false`; `at_risk` satisfies nothing; elevation claims ignored — "never identity, never authority").

Per [TECH_ID_07_PRECHECK.md](TECH_ID_07_PRECHECK.md) §4, TECH-ID-07 introduces an additive `devices/` module and a separate, memory/null `DeviceKeyModelStateV1` aggregate keyed by a caller-resolved `DeviceRootRefV1` (covering long-lived **and** ephemeral roots). This document states the invariants that module must uphold; the Device Authorization record stays authoritative in the Identity Firewall, and any root-side hint remains a **derived hint**, never a second source of truth.

## 1. Root / device confusion

The crown-jewel risk: collapsing the root↔device hierarchy so a subordinate device is mistaken for the root authority, or so a label / reference is mistaken for identity.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Device key treated as root identity** (a device grant read as the controller) | The hierarchy is fixed: a `LocalIdentityRoot` **authorizes** subordinate `DeviceAuthorization` records ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §11); the device record references exactly one root via `DeviceRootRefV1` and is never itself a root. Roots are PRIVATE and never peer-facing ([`identity-types.ts`](../../packages/identity/src/identity-types.ts)). Residual: the *relationship* is asserted locally, not proven — cryptographic subordination is Gate F. | **Design** (subordinate, one-root reference) + **Gate F** |
| **Root key copied to every device** (shared master secret) | No key bytes exist to copy: `keyMaterialState = "not_implemented_gate_f"` on every root ([`identity-types.ts`](../../packages/identity/src/identity-types.ts)), and ADR-0013 §11 requires **each future device to have independent key material** (Gate F) — never a shared root key. Residual: the "independent per-device key" property is a Gate F promise, not present today. | **Design** (no shared key model) + **Gate F** |
| **Device label treated as identity / authority** ("owner-laptop" reads as owner) | A device label is local presentation carrying `authority: "none"`; it is never an authorization input and is refused as a caller field by the boundary validator's `FORBIDDEN_FIELDS` discipline ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)). Residual: text can still *say* "owner" — but it is never read as authority. | **Design** (`authority: "none"` label) |
| **Device ID treated as a hardware identity** (device ref = serial/IMEI) | Device refs reuse the opaque-id discipline of [`identifiers.ts`](../../packages/identity/src/identifiers.ts) — bodies encode **no** hardware serial / model / MAC / timestamp; `hardwareId`, `deviceFingerprint`, and `attestation` are rejected caller fields ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)). Residual: an opaque ref still exists in local memory as a handle (endpoint scope, §8). | **Design** (opaque non-hardware ref) |
| **`RoomMemberRef` treated as a device credential** (room membership = a device) | `RoomMembershipRecordV1` carries **NO** device key / public key / fingerprint ([`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts)); ADR-0013 §11 keeps identity ≠ membership ≠ device, and the existing separation guards (`check:no-device-posture-or-governance-bypass`) hold the line. A membership can never stand in for a device authorization. Residual: none from this vector by construction. | **Design** (membership ≠ device) |
| **One device silently becomes the permanent primary authority** (first device self-promotes to root-equivalent) | The single bootstrap device record is `local_unverified` and `cryptographicallyAuthorized: false`, and creating it cannot be repeated to add further devices (§3); it grants no root authority and no device can promote itself. ADR-0013 §11: "a device cannot authorize another device unless explicitly allowed." Residual: on one device the bootstrap record *is* the only local context — a single-device install has no second authority to check it (§6). | **Design** (bootstrap is subordinate, non-authorizing) + **Inherent** (single-device install) |

## 2. Key-purpose confusion

A key slot is a typed, purpose-bound placeholder. Every path that lets one slot serve two purposes, or a purpose be read without checking, is an attack.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **One key reused for identity / messaging / rooms / backups** | The model is **one slot = one purpose** with `keyReuseAcrossPurposes: "forbidden"`; distinct signing / encryption / storage / linking purposes are separate slots that cannot be merged. Residual: this is a structural-metadata invariant, not a cryptographic domain-separation proof — real separation is Gate F. | **Design** (one slot = one purpose; reuse forbidden) + **Gate F** |
| **Signing key used as an encryption key** (cross-use of one slot) | Purpose is a required, typed field and is **immutable after activation**; a signing-purpose slot cannot be re-read as encryption. Residual: with `algorithm: "not_selected_gate_f"` there is nothing to actually sign or encrypt yet — the invariant guards a future, not a present operation. | **Design** (purpose immutable) + **Gate F** |
| **Key ID used without checking its purpose** (caller trusts the id, ignores the slot) | A slot reference resolves only through its typed record; consumers must read purpose from the record, and an unknown/absent purpose **denies** (fail closed). Residual: enforcement is a consumer obligation the module surfaces, not a wire guarantee. | **Design + Policy** (purpose-checked resolution; deny unknown) |
| **Algorithm / namespace confusion** (a slot claims a curve/suite it does not have) | No algorithm is selected: every slot carries `algorithm: "not_selected_gate_f"`, so no suite/namespace can be asserted or trusted; algorithm selection is Gate F. Residual: none to confuse today — restated as an honest gap. | **Design** (`algorithm: "not_selected_gate_f"`) + **Gate F** |
| **Future public material placed in the wrong slot** (a public key lands in a linking/backup slot) | `publicKey` / `privateKey` / `secretKey` are rejected caller fields ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)); no slot accepts key bytes in TECH-ID-07, and each slot's fixed purpose is the only home future material could occupy. Residual: correct slot-to-material binding must hold when Gate F ships — stated as a forward obligation. | **Design** (no key bytes accepted; fixed purpose) + **Gate F** |

## 3. Unauthorized device addition

Adding a device is the highest-authority operation in the model. Every path that mints an authorized device without the root's authority is an attack.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Current device authorizes another** without root authority | ADR-0013 §11: "a device cannot authorize another device unless explicitly allowed," and adding a device requires **explicit approval**. In TECH-ID-07 the add/link path is unavailable — `identity.device_key_future` is `effect: "future_gate"` ([`policyMatrix.ts`](../../packages/privacy/src/policyMatrix.ts)); a device cannot enroll a peer device. Residual: real cryptographic approval is Gate F — the guarantee is "no add path exists," not "add is cryptographically gated." | **Policy** (add is future-gated) + **Gate F** |
| **UI creates an authorized record directly** (client fabricates `cryptographicallyAuthorized: true`) | Authority/secret caller fields are never mass-assigned — `assertExactKeys` + `FORBIDDEN_FIELDS` reject `authenticated` / `trusted` / `verified` / `capabilities` ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)); the only creatable record is the `cryptographicallyAuthorized: false` bootstrap placeholder. Residual: none from this vector by construction. | **Design** (no caller-set authorization flag) |
| **DevicePosture grants authorization** (a "hardened" signal enrolls a device) | DevicePosture `mayOnlyTighten: true`, `trustedForElevation: false`, and is "never identity, never authority"; elevation claims are ignored and no provider is integrated ([`device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts)). Posture can restrict a device operation but never authorize one. Residual: none — posture cannot grant. | **Policy** (posture tightens only) |
| **Room-owner role authorizes a device** (RoomOS role grants device authority) | Role eligibility is "**necessary but never sufficient**" ([`membership/membership-roles.ts`](../../packages/rooms/src/membership/membership-roles.ts)) and grants only room capabilities; there is no room capability that mints an identity device authorization, and identity/membership stay separate ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §11). Residual: none from this vector by construction. | **Design** (RoomOS role ≠ device authority) |
| **Stale approval reused** (an old add token replays into a new authorization) | No add path and no approval token exist to replay (add is future-gated), and every accepted mutation is revision-checked (`IdentityLocalRevision` `expectedRevision`, [`identifiers.ts`](../../packages/identity/src/identifiers.ts)), so a stale expected revision is rejected. Residual: replay resistance for a *future* signed approval is Gate F. | **Policy** (no add path) + **Design** (revision-checked) + **Gate F** |
| **Test fixture becomes a production authorization** (a fixture device flagged authorized) | Production code may write only the non-authorized bootstrap placeholder; `cryptographicallyAuthorized: true` is not a producible state, and `PRODUCTION_ASSURANCE_STATES` bounds what can be written ([`identity-types.ts`](../../packages/identity/src/identity-types.ts)). A single-bootstrap guardrail (mirroring the existing identity bypass guards, precheck §5) blocks a second-device creation. Residual: fixtures must never be linked into production paths — a disclosed test-hygiene obligation. | **Design + Policy** (no producible authorized state; single-bootstrap guard) |

## 4. Device compromise and loss

A device is lost, stolen, or compromised. Every state that lets it stay trusted, keep broad scope, add more devices, or come back is an attack.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Compromised device stays active** (no way to flag it) | A `mark_compromised` transition moves the device record to a compromised-suspected state that **denies ordinary operations**, mirroring the root `compromised_suspected` lifecycle ([`identity-types.ts`](../../packages/identity/src/identity-types.ts)). Residual: the flag is a **local** state change with no remote effect — a compromised endpoint may ignore it (§6, §8). | **Design** (mark_compromised denies ops) + **External** |
| **Removal doesn't invalidate descriptors** (old snapshots stay usable) | `restrict` and lifecycle transitions **bump `IdentityLocalRevision`** ([`identifiers.ts`](../../packages/identity/src/identifiers.ts)); any snapshot carrying a stale `expectedRevision` fails the optimistic-concurrency check, invalidating cached descriptors. Residual: revision is local only — not tamper resistance and not distributed invalidation (§6). | **Design** (revision bump invalidates snapshots) |
| **Stale device keeps broad scope** (a lost device retains wide caps) | `restrict` **narrows** scope and caps and can never widen them (tighten-only, mirroring DevicePosture's `mayOnlyTighten`); scope reduction is monotonic per revision. Residual: narrowing is a **local** metadata effect, not a key revocation — the future key stays intact until Gate F. | **Design** (restrict narrows only) + **Gate F** |
| **A stale device adds more devices** (compromised device enrolls peers) | The device add/link path is unavailable (`identity.device_key_future` = `future_gate`, [`policyMatrix.ts`](../../packages/privacy/src/policyMatrix.ts)); no device — compromised or not — can enroll another. Residual: none from this vector by construction (add is deferred). | **Policy** (`device.add` future-gated) |
| **History access survives device removal** (removed device still reads the past) | ADR-0013 §11: "history access and device authorization are **separate** decisions"; revoking a device changes device authorization state and does not silently confer or preserve history access. Residual: on a compromised endpoint the local data the device already holds is outside module reach (§8). | **Design** (history ≠ authorization) + **External** |
| **Revoked device reactivates** (tombstone resurrected) | Revoke is a **terminal tombstone** and marks the device's key slots unusable, mirroring the terminal `removed_tombstone` membership transition (`assertMembershipTransition` refuses any transition out of a tombstone, [`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts)); a revoked record cannot transition back to active. A "new" device is a new record, never a revival. Residual: terminality is **local** — a peer or offline copy is not reached (§6). | **Design** (revoke is terminal + slots unusable) + **Gate H** |
| **Root compromise confused with device compromise** (one taints/absolves the other) | Root and device compromise are modeled **separately** — the root lifecycle (`compromised_suspected` / `revoked_tombstone`, [`identity-types.ts`](../../packages/identity/src/identity-types.ts)) is distinct from a device record's lifecycle; marking one does not auto-mark the other, and a device event does not revoke the root. Residual: the correct blast-radius of a *real* key compromise is a Gate F property, not decided here. | **Design** (separate root/device compromise) + **Gate F** |

## 5. Correlation and metadata

A device model is a rich correlation surface: hardware ids, reused device ids, and a device list can re-link roots, rooms, and relationships. Every leak is an attack.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Hardware serial used as the device ID** (device ref = serial/MAC/IMEI) | Device refs are opaque and encode **no** hardware data; `hardwareId` / `deviceFingerprint` are rejected caller fields, and no code collects serial/model/MAC/advertising ids ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts), precheck §3). Residual: none from this vector by construction. | **Design** (opaque non-hardware ref) |
| **A stable device ID reused across roots** (one ref links a user's independent roots) | Refs are **per-context** (per `DeviceRootRefV1`, covering long-lived and ephemeral roots) and encode no root material, echoing "no single device identifier shared across unrelated contexts by default" ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §11). Independent roots stay isolated (ADR §5). Residual: the local memory holds the ref→root mapping (endpoint scope, §8). | **Design** (per-context opaque ref) |
| **Device name / model uploaded** (label or model leaves the device) | Labels are **local-only**, never networked and never logged; there is no export/sync path in TECH-ID-07 (memory/null, precheck §4). Residual: label text lives in local memory (§8). | **Design** (label local-only; no export) + **Gate E** |
| **Device list exposes the persona / relationship / room graph** | A device summary is **redacted**: strict modes omit ref / label / scope / count / timestamps, mirroring the redacted ephemeral summary discipline ([`ephemeral/ephemeral-types.ts`](../../packages/identity/src/ephemeral/ephemeral-types.ts) `EphemeralIdentitySummaryV1`) and the redacted membership views ([`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts)). A device record carries no relationship/room/persona ids. Residual: a local caller with full-detail rights still sees its own device list (endpoint scope, §8). | **Design** (redacted summaries; no graph fields) |
| **Revocation leaks timing / count** (revoke reveals how many devices exist / when) | Revocation results are content-free and redacted — no ref, no remaining-device count, no trusted timestamp (only `local:`-prefixed local labels, cf. [`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts)); errors are content-free `IdentityValidationError` ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)). Residual: a local observer of the process sees local effects (§8). | **Design** (content-free errors + redacted audit) |
| **Labels surface in logs / notifications** | Labels are `authority: "none"` local metadata that must never be networked or logged; TECH-ID-07 emits no telemetry and no notification carries a device label. Residual: OS-level notification mirroring / logging is external (§8). | **Design** (labels never logged/networked) + **External** |

## 6. Multi-device and synchronization limits

TECH-ID-07 is single-vault and local. The threats here are about *pretending it is distributed* — treating a local device state as a synchronized, globally revocable fact. These are honest limitations (Gate H).

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Local revocation described as global** (revoke on one device implies revoke everywhere) | Revocation is a **local** terminal tombstone with no wire effect; "Local identity state is authoritative only for the current local vault; no distributed consistency or revocation is claimed" ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §19, Gate H). Residual: other devices are not reached — stale trust can persist elsewhere. | **Gate H** (no distributed revocation) + **Inherent** |
| **Remote device list assumed fresh** (the list reflects all devices everywhere) | There is no synchronization of device-auth state (ADR §19 lists device-auth under Gate H); the aggregate is one local memory/null vault. Residual: a device enrolled/removed elsewhere is simply not represented locally. | **Gate H** (sync deferred) |
| **Offline device uses old state** (a disconnected device acts on stale authorization) | No causal ordering and no shared clock exist; `IdentityLocalRevision` is explicitly **not** a vector clock and **not** synchronization ([`identifiers.ts`](../../packages/identity/src/identifiers.ts)). Residual: an offline device's local view can diverge arbitrarily — disclosed. | **Gate H** + **Inherent** (distributed state) |
| **Device change races with room operations** (concurrent device + room mutation) | Within one vault, revision checks serialize local mutations; **across** devices there is no coordination, so a device change and a room op on separate devices are not ordered. Residual: cross-device races are unresolved by design until Gate H. | **Gate H** (no cross-device ordering) |
| **A synchronized device epoch is assumed** (callers expect a shared device generation) | There is **no** synchronized device epoch; the only epoch present is the **ephemeral process epoch** (`processBinding`, current-process only, [`ephemeral/ephemeral-types.ts`](../../packages/identity/src/ephemeral/ephemeral-types.ts)), which is not a cross-device generation counter. Residual: any code expecting a global device epoch is mistaken — stated, not solved. | **Gate H** + **Inherent** |

## 7. Recovery and backup

Recovery is the classic backdoor into a device model. TECH-ID-07 ships **no** backup, export, or recovery — this section states that plainly rather than implying a safe path.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **Root / device material copied through cloud backup** | No export/backup path exists: roots and device records are memory/null (`persistenceClass: "memory_only" \| "null"`, [`identity-types.ts`](../../packages/identity/src/identity-types.ts)), key material is `not_implemented_gate_f`, and `storage.write.key_material` is `future_gate` ([`policyMatrix.ts`](../../packages/privacy/src/policyMatrix.ts)). Residual: OS-level device backups of process memory are external (§8). | **Design** (no export; memory/null) + **External** |
| **Recovery restores revoked devices** (a restore resurrects a tombstoned device) | There is no recovery mechanism in TECH-ID-07; the Recovery Kit is an architecture concept only (no formats/algorithms, [ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §13, gated by a dedicated recovery threat review). Revoke is terminal (§4). Residual: recovery semantics are a **future** design that must preserve terminality — a forward obligation, not a present guarantee. | **Gate F / Future** (recovery deferred) + **Design** (revoke terminal) |
| **Backup merges device identities** (restore fuses two roots' devices) | No backup/merge path exists; independent roots stay isolated (ADR §5), and a device record references exactly one root. Residual: any future recovery must not merge device sets across roots — disclosed forward obligation. | **Design** (per-root isolation; no merge path) + **Future** |
| **History transfer implies authorization** (restoring history re-authorizes a device) | ADR-0013 §11 keeps history access and device authorization **separate**; there is no transfer path that could conflate them today. Residual: future recovery/history features must keep the decisions separate — stated, not solved. | **Design** (history ≠ authorization) + **Future** |
| **Device authorization hidden inside recovery** (a restore silently enrolls a device) | No recovery flow exists to smuggle an authorization through; the add path is future-gated (§3) and a dedicated recovery threat review is a prerequisite gate (ADR §24). Residual: this is a design constraint on future recovery, not a mitigation of a present feature. | **Future / Gate F** (recovery + add both deferred) |

## 8. Endpoint boundary

The endpoint is outside the module's trust boundary. TECH-ID-07 states this plainly rather than implying custody it does not provide.

| Attack | Why it fails / residual | Control |
| --- | --- | --- |
| **A future key sits on a compromised OS** (malware reads it) | TECH-ID-07 holds **no** key bytes to steal (`keyMaterialState: "not_implemented_gate_f"`); when Gate F ships, key custody on the endpoint is an **external** Secure Device / Endpoint Defense concern ([ADR-0012](../adr/ADR-0012-endpoint-defense-layer.md)), not defended by core. Residual: total on a compromised endpoint — malware sees what the user sees. | **External** (endpoint) + **Gate F** |
| **Malware steals future material** (exfiltration of key/authorization bytes) | Nothing to exfiltrate today; the module claims no anti-exfiltration, and secure storage/custody is externalized. Residual: future material's protection is entirely an external-provider property. | **External** (Secure Device) + **Gate F** |
| **Attestation absent** (no proof the device is genuine/uncompromised) | There is **no** attestation: `attestation` is a rejected caller field ([`identity-validation.ts`](../../packages/identity/src/identity-validation.ts)) and the Device Passport (future signed root↔device relationship) is TECH-ID-08 and "implies **no** hardware attestation" ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §11). Residual: device genuineness is unprovable in TECH-ID-07 — stated, not solved. | **Design** (no attestation accepted) + **TECH-ID-08 / Gate F** |
| **High DevicePosture mistaken for safe custody** (a "hardened" reading treated as key safety) | DevicePosture is an environment attribute that "may only tighten" and is "never identity, never authority"; no provider is integrated (`providerIntegrated: false`) and elevation claims are ignored ([`device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts)). A posture reading is never evidence of key custody. Residual: UX must not render posture as custody — disclosed obligation. | **Policy** (posture ≠ custody) + **External** |
| **Secure Device provider compromised** (the external custodian is subverted) | Secure Device is a separate, externalized project (`check:no-secure-device-core-implementation`, [`policy-composition/device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts)); core makes no claim about the provider's integrity and does not depend on it for the device model's local correctness. Residual: a subverted provider is outside core's defense — disclosed. | **External** (Secure Device) |

## Explicitly NOT provided (read this)

**Stated plainly, without hedging:**

- **No real key security.** TECH-ID-07 stores no key bytes, derives no keys, and performs no cryptographic operation (`keyMaterialState: "not_implemented_gate_f"`, `algorithm: "not_selected_gate_f"`). It is a local, non-cryptographic, metadata-only device model.
- **No cryptographic authorization.** A Device Authorization record is `cryptographicallyAuthorized: false`. Nothing signs a root↔device relationship, and nothing proves a device is subordinate to a root. Signed authorization is Gate F.
- **No remote revocation.** Revocation is a local terminal tombstone with no wire effect. There is no distributed device revocation, no "remove on the other device," and no synchronized device state. Reconciliation is Gate H.
- **No hardware binding.** Device references are opaque and encode no serial / model / MAC / IMEI / advertising id; `hardwareId` and `deviceFingerprint` are rejected. A device is not bound to physical hardware.
- **No endpoint guarantee.** Key custody, exfiltration resistance, and attestation are external (Secure Device / Endpoint Defense) or absent. Memory/null retention does not stop a compromised OS, malware, backups, or crash dumps.
- **Not safe for real secrets.** Until reviewed identity/crypto (Gate F) and a Secure Device integration ship, the device model must not be treated as a safe home for information whose exposure would cause real harm.

**And, stated as invariants:**

- **A DeviceKey is not DevicePosture.** A key slot is an identity-owned typed placeholder; DevicePosture is a separate RoomOS environment attribute that may only tighten ([`device-posture.ts`](../../packages/rooms/src/policy-composition/device-posture.ts)).
- **DevicePosture is not identity**, and a high posture reading is never proof of key custody, device genuineness, or authorization.
- **A device is not a person.** A device authorization is a subordinate grant, never a personhood or real-world identity claim ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §4, §11).
- **A device reference is not a hardware identifier.** It is an opaque per-context handle that encodes no hardware data ([`identifiers.ts`](../../packages/identity/src/identifiers.ts)).
- **A device label is not authority.** Labels carry `authority: "none"` and are never an authorization input.
- **A local authorization record is not a cryptographic proof.** `cryptographicallyAuthorized: false`; the subordination is asserted locally, not proven (Gate F).
- **RoomOS membership is not device authorization.** `RoomMembershipRecordV1` carries no device key and a room role is "necessary but never sufficient" ([`membership/membership-types.ts`](../../packages/rooms/src/membership/membership-types.ts), [`membership/membership-roles.ts`](../../packages/rooms/src/membership/membership-roles.ts)).
- **Device revocation is not remote erasure.** A revoke tombstone is local; it makes no remote or forensic-erasure claim ([ADR-0013](../adr/ADR-0013-identity-firewall-architecture.md) §9, §19).
- **Deferred Gates E / F / H + TECH-ID-08.** Wire/remote exchange (E), crypto / real keys / signatures / device-authorization proofs / verification fingerprints (F), sync / distributed device revocation (H), and the Device Passport (TECH-ID-08) are all out of scope for TECH-ID-07.

## Top threats — indicative severity / likelihood and resolving control

Severity and likelihood are **indicative pre-implementation judgments**, not measurements — they rank where design and test attention are most needed. "Resolved / bounded by" names the control (Design / Policy / Gate) that carries each item; none is claimed fully solved.

| # | Threat | Severity | Likelihood | Resolved / bounded by |
|---|--------|----------|------------|-----------------------|
| 1 | A device silently becomes / is read as the root authority, or a root key is treated as shared across devices — (1) | Critical | Low | **Design** (subordinate one-root reference; no shared-key model) + **Gate F** |

| 6 | Device list / summary leaks the persona / relationship / room graph, or revocation leaks count/timing — (5) | High | Medium | **Design** (redacted summaries; content-free errors; no graph fields) |
