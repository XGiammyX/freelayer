# TECH-23 — Sensitive Room Admission + Secure Device Integration Contract Threat Model

_Scope: sensitive-room admission gate, posture assessment contract, Secure Device provider port, transient admission session, provider failure/recovery. Extends [TECH_22_POLICY_COMPOSITION_GOVERNANCE_THREAT_MODEL.md](TECH_22_POLICY_COMPOSITION_GOVERNANCE_THREAT_MODEL.md)._

> [!IMPORTANT]
> Read the [Honest limitations](#honest-limitations) section first. FreeLayer core is only the future RATS **Relying Party**: **no Secure Device provider is integrated** (a deterministic Null provider is the only implementation). Posture can never exceed `unverified` in core; `at_risk` can only tighten. Secure Device / Endpoint Defense is a **separate external project**, not part of this repository. This contract defines the narrow boundary that project would one day implement — it does not add endpoint protection today.

## What this contract is (and is not)

Admission is an **ADDITIONAL local gate** resolved immediately before authorization/execution. It NEVER replaces membership, capability, or `PolicyDecision`; a positive admission confers no authority on its own. The gate is deterministic, fail-closed, and REDACTED — its output is codes only (`outcome`, `reasonCodes`), never content, raw evidence, or identifiers.

Provenance for a posture assessment is enforced by a **module-private `WeakSet` registry** (`issuedAssessments`), the same pattern used for `PolicyDecision`. Only objects this module actually issued pass `isAcceptedDevicePostureAssessmentV1`. This is genuinely unforgeable within the JS object-capability model — no other code holds a reference to the registry, so no other code can add to it — but it is **NOT cryptographic (Gate F)**: it does not defend against same-realm reflection on a real assessment, and it makes no attestation claim. Freshness is **current-process-only**: there is no trusted clock, nonce, or epoch, so there is **no replay-resistance claim**.

## Provider impersonation

Arbitrary code claims to be a Secure Device provider; the UI constructs a `high_assurance` status; a test provider (`ready_future`) enters a production path; a hand-crafted object mimics an accepted assessment structurally; a caller marks a provider "ready" to unlock content; a forged status asserts `trustedForPostureElevation`. **Mitigation:** `buildProductionProviderStatusV1` is fail-closed — only `not_integrated`/`unavailable`/`failed` survive; any `initializing_future`/`ready_future`/`degraded_future` lifecycle is coerced to `unavailable` (reason `future_lifecycle_coerced_unavailable`). `providerTrustedForElevationV1` is **always `false`** in core: `trustedForPostureElevation` is structurally typed `false`, so the `trusted && lifecycle === "ready_future" && canAssessPosture` conjunction can never hold, and it is kept as the single testable source of truth for that honest answer. `isAcceptedDevicePostureAssessmentV1` requires BOTH module-issued provenance (`issuedAssessments.has`) AND structural validity (`isStructurallyValidAssessmentV1`), so a structurally perfect look-alike is still rejected for lack of provenance. The only production factory (`resolveDevicePostureAssessmentV1`) resolves through the fail-closed TECH-22 resolver with `providerIntegration: "not_integrated"`, yielding `unverified` (default) or `at_risk` (untrusted tightening) and nothing higher — there is no production `basic+`/`high_assurance` factory. In the admission resolver, an unaccepted assessment collapses `effectivePosture` to `unverified`, and any room requiring `basic+` denies content (`deny_minimum_posture` / `deny_posture_unverified`). **Residual risk:** a same-realm attacker who obtains a reference to a genuinely issued assessment could reflect on it — this is the stated non-cryptographic (Gate F) boundary.

## Evidence leakage

Raw attestation or a measurement log enters core through the assessment; hardware/OS identifiers (serial, IMEI, `androidId`, build fingerprint) are logged; the installed-app inventory is exposed; evidence lands in a crash report; assessment history accumulates into a tracking signal. **Mitigation:** `FORBIDDEN_ASSESSMENT_FIELDS` enumerates evidence-/identifier-bearing keys (`rawEvidence`, `attestation`, `measurementLog`, `deviceId`, `serial`, `imei`, `androidId`, `osBuild`, `installedApps`, `assessmentHistory`, `nonce`, `signature`, `certificateChain`, …) and `isStructurallyValidAssessmentV1` rejects any object carrying one on sight — an evidence-bearing object cannot become a policy input. The assessment shape carries structural literals asserting content-freedom: `rawEvidenceIncluded`/`deviceIdentifiersIncluded`/`deviceManagementPerformed`/`telemetryEmitted` are all typed `false`, and validation rejects any deviation. The provider capabilities declare `rawEvidenceExport: "forbidden"`, `deviceManagement: "forbidden"`, `telemetry: "forbidden"` — no export/management/telemetry surface exists to leak through. Admission decisions and session errors are content-free (`redacted: true`, codes only), so nothing evidence-shaped can reach a log or crash report. The assessment is TRANSIENT and never persisted, so no history is retained to become tracking data. **Residual risk:** what a future external provider does with real evidence on the device is outside this contract; core simply refuses to accept or store any of it.

## Freshness & replay

A stale assessment is reused to keep content open; a lower revision (rollback) is silently accepted; an admission granted before a posture change survives it; a provider failure is ignored and the session stays open; a local timestamp is treated as a trusted freshness proof. **Mitigation:** `checkAssessmentFreshnessV1` returns `fresh: false` for `stale` and for `revision_rollback` (a revision below the last one seen this process) — rollback is rejected, never silently accepted — and treats an unknown freshness value as not fresh. `classifyProviderTransitionV1` maps environment changes to explicit effects: `reject_rollback` on a lower revision, `invalidate_admission` on `at_risk`/`unavailable`/`failed`/`stale`, `reduce_to_unverified` on `not_integrated`. The transient session (`assertSensitiveRoomSessionCurrentV1`) is bound to the exact revisions it depended on and fails closed (`SensitiveRoomSessionStaleError`) on ANY change — assessment/policy/membership/mode revision, action, `at_risk` posture, provider unavailable/failed, or stale freshness — and is never silently refreshed or restored. `DEVICE_POSTURE_FRESHNESS_POLICY_V1` states the honest posture explicitly: `trustedClockAvailable: false`, `nonceVerificationAvailable: false`, `epochVerificationAvailable: false`, `crossRestartValidity: false`; the session carries `persistence: "forbidden"` and `crossRestartValidity: false`. Recovery (`PROVIDER_RECOVERY_REQUIREMENT_V1`) requires a fresh assessment, a fresh admission, and fresh authorization, and structurally `restoresOldSession: false` / `autoRevealsContent: false` / `emitsBackgroundNotification: false`. **Residual risk (stated):** freshness is current-process-only; there is **no replay-resistance claim** and no cross-restart validity — a local timestamp is diagnostic only, never trusted.

## Admission bypass

A room opens content before the posture check runs; a redacted-summary permission is treated as a content permission; a room's minimum posture is ignored; a ScreenShield/managed-Bunker requirement is silently downgraded; an `at_risk` device is admitted anyway; posture is used to bypass membership or `PolicyDecision`. **Mitigation:** `resolveSensitiveRoomAdmissionV1` is a single fail-closed resolver evaluated in a fixed 14-step order (validate action → Privacy Mode/emergency → lifecycle → membership/capability screen → redacted-summary branch → manage branch → content: `at_risk` deny → provider lifecycle → freshness → minimum posture → protected presentation → Bunker session → future-gated actions → permit), with **deny-overrides** throughout and an unclassified action failing closed (`deny_policy`/`unknown_action`). The redacted summary is a distinct outcome (`allow_redacted_summary_only`) that only ever yields a `redacted_summary_only` session and NEVER confers content — content and summary are separate outcomes with separate sessions. Minimum posture is enforced by `devicePostureSatisfiesMinimumV1` (explicit ranks, never lexical); any room requiring `basic+` denies because no trusted provider exists (`no_provider_for_required_posture` → `deny_minimum_posture`). A future-required protected presentation denies (`deny_protected_presentation_unavailable`), and a `managed_bunker_future_required` requirement denies (`deny_bunker_session_unavailable`) — there is no silent downgrade path. `at_risk` denies all content and management actions (`deny_posture_at_risk`). Critically, admission is an ADDITIONAL gate resolved before authorization/`PolicyDecision`, never a replacement: the membership/capability step only screens out obviously-revoked local authority and cannot GRANT — real authorization remains separate and downstream. **Residual risk:** enforcement is local-only; a non-compliant client that skips the gate entirely is an accepted limitation (Gate F/G/H), consistent with TECH-20/21/22.

## Endpoint limits (honest residual risk — not mitigable in core)

These attacks cannot be mitigated by an admission gate or a posture contract; they belong to the endpoint, the OS, or the physical world, and are listed so the contract does not overclaim:

- **Provider / OS / kernel compromise** — root/jailbreak/kernel malware defeats app-level gates entirely; a compromised (future) provider could misreport, which is exactly why core trusts none.
- **Camera pointed at the display** — an external camera records rendered content; unsolvable in software.
- **Hostile keyboard / Accessibility service** — legitimate OS input/observation channels that read the screen and inject input; indistinguishable from assistive technology the platform must support.
- **External capture** — casting, remote desktop, screen sharing, OS "recall"-class recorders capturing the rendered frame.
- **Physical attacker** — shoulder-surfing, an unlocked device, or coercion.

Admission decides whether to REVEAL content; it cannot police what happens to pixels after reveal. This mirrors [THREAT_MODEL.md — Endpoint display/capture threats](../THREAT_MODEL.md) and [ENDPOINT_DEFENSE_MODEL.md](../ENDPOINT_DEFENSE_MODEL.md).

## Honest limitations

**Stated plainly, without hedging:**

- **No endpoint guarantee.** A compromised device, OS, or kernel defeats this contract; admission gates reveal, it does not harden the endpoint.
- **No spyware-proof guarantee.** Keyloggers, screen recorders, and Accessibility abuse operate through channels core cannot see or block.
- **No capture-proof guarantee.** Nothing here prevents screenshots, screen recording, casting, or a camera aimed at the display.
- **No trusted provider exists.** Core is only the future RATS Relying Party; the sole implementation is a deterministic Null provider. `providerTrustedForElevationV1` is always `false` and posture never exceeds `unverified` (`at_risk` only tightens).
- **No evidence verification exists.** There is no attestation, no cryptographic verification, no trusted clock/nonce/epoch (Gate F). Provenance is object-capability, not cryptographic.
- **Not safe for real secrets.** Until a reviewed external Secure Device / Endpoint Defense integration ships, sensitive rooms must not be treated as a safe place for information whose exposure would cause real harm.

Secure Device / Endpoint Defense is a **separate external project**. This contract is the honest, fail-closed boundary it would implement — nothing in core today provides device attestation or endpoint protection.
