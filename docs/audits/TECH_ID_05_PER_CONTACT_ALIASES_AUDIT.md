## Two record classes (the deliverable)

TECH-ID-05 introduces exactly two local, relationship-scoped presentation records in a **separate** aggregate `ContactAliasStateV1` (`persistenceClass: "memory_only" | "null"`) that references a relationship **by id** ‚Äî the relationship record is never rewritten ([`alias-types.ts`](../../packages/identity/src/aliases/alias-types.ts)). Neither is identity, verification, authentication, a public username, or a cryptographic identifier.

### Pairwise presentation alias

`PairwisePresentationAliasV1` is a relationship-scoped, persona/root-bound, local-only display name. It fixes `sharingState: "not_shared_tech_id_05"`, `authenticatedBinding: "not_implemented_gate_f"`, `remoteUpdateState: "not_implemented_gate_e_f"`, and carries a `ContactAliasSecurityAssessmentV1 { impersonationSafe: false, verifiedIdentity: false, confusableAssessment: "not_evaluated" | "future_unicode_security_gate" }`. Lifecycle `draft_local ‚Üí active_local_unshared ‚Üí retired_tombstone` (terminal), enforced by an explicit transition table ([`alias-lifecycle.ts`](../../packages/identity/src/aliases/alias-lifecycle.ts)). At most **one** active (non-retired) alias per relationship is enforced in the reducer (`hasActiveAliasForRelationship`, plus a second `duplicate_active` guard on activate). It is never transmitted, negotiated, or authenticated to the peer in this task.

### Local peer label

`LocalPeerLabelV1` is a private note-to-self, fixing `visibility: "local_only"`, `peerShared: false`, `authority: "none"`, `verificationEvidence: false`. It is **NEVER** sent to the peer under any mode. Lifecycle `active_local_private ‚Üí cleared_tombstone`; **clear removes the record entirely** (no retained text) and **replace overwrites in place** (no old plaintext retained) ([`alias-reducer.ts`](../../packages/identity/src/aliases/alias-reducer.ts)). At most one active label per relationship. A local peer label is **not** verified peer identity; the display context surfaces verification state separately and never as the label.

## Text normalization (conservative, honest)

`normalizeContactAliasDisplayTextV1` ([`alias-normalization.ts`](../../packages/identity/src/aliases/alias-normalization.ts)) applies **one** transform: `input.normalize("NFC").trim()` (NFC, **not** NFKC ‚Äî compatibility normalization can alter visible meaning), then rejects dangerous code points via `isRejectedCodePoint`: C0 controls `U+0000..U+001F` (incl. **NUL**), DEL + C1 `U+007F..U+009F`, line/paragraph separators `U+2028`/`U+2029`, bidi overrides `U+202A..U+202E`, bidi isolates `U+2066..U+2069`, zero-width space `U+200B`, word joiner `U+2060`, and BOM/ZWNBSP `U+FEFF`. **ZWJ `U+200D` and ZWNJ `U+200C` are retained** (required for e.g. Persian, Devanagari, and emoji ZWJ sequences). Bounds: `CONTACT_ALIAS_LIMITS_V1` ‚Äî scalar length 1..64, ‚â§256 UTF-8 bytes; raw input is discarded (`originalInputState: "discarded_after_validation"`). It does **not** case-fold, transliterate, or block whole scripts, and is **not** a UTS #39 confusable detector ‚Äî normalization is **not** proof an alias is safe.

## Local reuse warning (privacy nudge only)

`assessContactAliasReuseV1` ([`alias-correlation.ts`](../../packages/identity/src/aliases/alias-correlation.ts)) reports whether the same *normalized* value is already used by an active presentation alias in another relationship of the current vault. It is a **WARNING only**: it never denies, never merges relationships, and returns `otherRelationshipCountExposed: false` / `relationshipIdsExposed: false` ‚Äî no ids, no counts, no telemetry. A unique value is **not** proof of unlinkability; a reused value is **not** proof identities are linked. Cross-root (long-lived ‚Üî ephemeral) reuse produces the same warning without revealing the root mapping.

## Lifecycle & rotation

Transition tables are explicit and terminal-is-terminal; unknown/self transitions throw `ContactAliasLifecycleError` ([`alias-lifecycle.ts`](../../packages/identity/src/aliases/alias-lifecycle.ts)). **Rotation** (`identity.alias.presentation.rotate`) is one pure reducer step that retires the old alias and creates a **new** `active_local_unshared` record under the same relationship id, revision-checked (`revision_mismatch`) and ref-matched. It preserves relationship continuity ‚Äî `ContactAliasRotationResultV1 { relationshipContinuityPreserved: true, trustStateChanged: false, blockStateChanged: false, remoteCopiesAffected: false, authenticatedRemoteUpdatePerformed: false }` ‚Äî matching ADR-0013 ¬ß9 ("block state survives display-alias changes"). Block/trust live on the relationship, not the alias; rotation cannot touch them, delete remote copies, or perform an authenticated update. A retired alias cannot reactivate; a new presentation is a new record.

## Command validation (fail-closed)

`validateContactAliasCommandV1` ([`alias-validation.ts`](../../packages/identity/src/aliases/alias-validation.ts)) is a closed discriminated union of **7** commands (create / activate / rotate / retire; local_peer_label set / replace / clear ‚Äî [`alias-commands.ts`](../../packages/identity/src/aliases/alias-commands.ts)); there is **no** generic alias/profile patch. It enforces exact schema version, exact discriminator, and exact keys; rejects dangerous keys (`__proto__`/`prototype`/`constructor`) and a `FORBIDDEN_FIELDS` set covering authority/verification/public-directory/secret caller fields (`verified`, `trusted`, `authenticated`, `public`, `global`, `searchable`, `uniqueUsername`, `handle`, `phone`, `email`, `did`, `publicKey`, `privateKey`, `relationshipKey`, `rootPublicId`, `devicePosture`, `roomRole`, `isOwner`, `isAdmin`, `capabilities`, `screenShieldActive`, `peerShared`, `securityAssessment`, `sharingState`, `authenticatedBinding`). Text fields are normalized through the canonical normalizer (raw input never appears in errors); the caller may not set security-assessment or sharing/authentication state; every non-create update carries the expected current revision.

## Policy + PolicyDecision integration (exact-scope)

`resolveContactAliasPolicyV1` ([`alias-policy.ts`](../../packages/identity/src/aliases/alias-policy.ts)) is pure, deterministic, fail-closed, strictest-wins across the seven privacy modes over **9** operations. Bunker/Emergency deny expansive writes; Ghost also denies local peer labels; Emergency permits only restrictive ops (`presentation.retire`, `local_peer_label.clear`, `display_context.read`); strict modes force `null` retention and suppress exact timestamps. `networkSharingAllowed`, `publicDirectoryAllowed`, `globalUsernameAllowed`, `authenticationAvailable`, `cryptographicBindingAvailable`, `persistentPlaintextAllowed`, `notificationAllowed`, `telemetryAllowed`, and `aiAllowed` are **structurally `false` in every mode**. The pipeline `applyContactAliasCommandV1` ([`alias-pipeline.ts`](../../packages/identity/src/aliases/alias-pipeline.ts)) maps each operation onto one of **7 exact `identity.alias.*` side-effect scopes** and requires an authentic, allowed, **exact-scope** `PolicyDecision` ‚Äî a forged, wrong-scope, or denied decision is rejected **before** mutation. An alias id, display text, relationship id, RoomOS role, or DevicePosture never authorizes; alias-text collision never grants authority; a display-context read requires its own `display_context.read` decision.

## Pure reducer + memory/null repositories

`reduceContactAliasCommandV1` ([`alias-reducer.ts`](../../packages/identity/src/aliases/alias-reducer.ts)) is pure and deterministic: validate-before-mutate; injected ids (`ContactAliasGeneratedIdsV1`) + injected clock + a validated `ContactAliasRelationshipRefV1` only; no storage/network/notification/AI/crypto/randomness; source state/command never mutated; revisions increment by one; failure yields no partial state. Ids are opaque, injected, and non-derived (`alias-‚Ä¶` / `plabel-‚Ä¶`; `^[a-z0-9][a-z0-9_-]{2,127}$`, no `@`; encode no display text, root/persona/relationship id, phone/email/username, timestamp, room id, device info, or key material ‚Äî [`alias-identifiers.ts`](../../packages/identity/src/aliases/alias-identifiers.ts)). `InMemoryContactAliasRepositoryV1` and `NullContactAliasRepositoryV1` ([`alias-repository.ts`](../../packages/identity/src/aliases/alias-repository.ts)) are memory-only / null ‚Äî no disk, browser storage, database, or encrypted storage ‚Äî with deep defensive `structuredClone`, no global singleton, no serialized snapshot; a `null`-classed state is discarded rather than stored. Every repository operation re-checks `isAuthentic`: an authentic decision, `verdict === "allowed"`, and a `sideEffect` beginning with `identity.alias.` ‚Äî a non-alias scope is refused (`decision_mismatch`).

## Redacted errors / audit / display context

`ContactAliasError` ([`alias-errors.ts`](../../packages/identity/src/aliases/alias-errors.ts)) carries only a stable code from a 13-member union (`validation`, `normalization`, `policy_denied`, `decision_mismatch`, `not_found`, `revision_mismatch`, `lifecycle`, `relationship_mismatch`, `duplicate_active`, `transfer_forbidden`, `remote_sharing_unavailable`, `authentication_unavailable`, `local_peer_label_visibility`) plus the single static `SAFE_ALIAS_REJECTION` string ‚Äî never alias text, a label, a relationship/root/persona id where avoidable, a command payload, collision details, or a leak sentinel. `buildContactDisplayContextV1` ([`alias-summary.ts`](../../packages/identity/src/aliases/alias-summary.ts)) returns trust/assurance **separately** from alias text, never a generic "verified contact" flag, and never a local peer label to any peer/room surface; it fixes `aliasVerified: false`, `realWorldIdentityVerified: false`, `cryptographicBindingAvailable: false`, and redacts the relationship id + labels in strict modes. Audit events are `redacted: true` (category + outcome + reason code only).

## Secure Device separation

Preserved. The alias module imports only `@freelayer/privacy` (+ transitively `@freelayer/security` via identifiers) and sibling identity types; it imports **no** Secure Device / DevicePosture / ScreenShield / attestation implementation, no UI, no transport, and no crypto. DevicePosture remains tighten-only in policy and a rejected caller field in validation; a device posture or room role can restrict alias operations but can never validate, sign, or bless alias text. Endpoint protection (screenshots, task-switcher previews, secure input, memory hardening) is the **external** Secure Device / Endpoint Defense project's concern; the core keeps only contracts/policy inputs/honest disclosures.

## Known limitations / deferred gates

Stated plainly. An alias is **not** identity, **not** verification, **not** authentication, **not** a public username, **not** a cryptographic identifier, and does **not** prove a person controls a key. A local peer label is **NEVER** sent to the peer. **DevicePosture is not identity; room membership is not identity proof; a `RoomMembershipId`/room role is not an alias authority.** Normalization is conservative baseline hygiene and is **not** proof an alias is safe ‚Äî it is **not** a UTS #39 confusable/homograph detector, and a mixed-script or look-alike name can still be accepted; security-sensitive UI must never rely on alias text alone. The local reuse warning detects only identical *normalized* text in the current local vault. There is **no** remote alias deletion, sharing, or synchronization; anything a future exchange delivered would persist beyond local control. Relationship-scoping and opaque non-derived ids reduce alias-driven correlation, but this is structural/organizational, not cryptographic ‚Äî reused text/avatars/behavior or a compromised endpoint can still correlate contexts. Aliases are **memory/null only and not safe for real secrets**. Deferred: cryptographic alias authenticity / key material (**Gate F**), wire formats / remote alias exchange (**Gate E**), the full Identity Firewall facade wiring RoomOS‚Üîalias results (**Gate G**), and synchronization / distributed alias revocation (**Gate H**). Verified identity, device passports, recovery, identity keys, key transparency, and DID are **not** implemented in this task ‚Äî they are named here only to negate them.

## Verdict

**Complete.** The local, relationship-scoped, non-cryptographic per-contact alias layer is implemented as a separate aggregate of two presentation classes (pairwise presentation alias + local peer label), with conservative NFC normalization + dangerous-control rejection, a privacy-only reuse warning, explicit terminal lifecycles, continuity-preserving rotation, a fail-closed command validator, exact-scope policy gating enforced at the pipeline and again at the repository, a pure side-effect-free reducer, memory/null repositories, redacted errors/audit/display contexts, +15 Policy Matrix rows (6 structural denies/future-gates), a dedicated static guardrail, and honest documentation. An alias is never identity, verification, authentication, a public username, or a cryptographic identifier; a local peer label is never peer-facing; DevicePosture and room role never authorize. Enforcement note (see [boundary audit](CONTACT_ALIAS_BOUNDARY_AUDIT.md)): the sdk/core Identity Firewall facade that maps RoomOS ‚Üî alias display results is a later task (Gate G). Deferred: crypto/authenticity (Gate F), wire/exchange (Gate E), sync/distributed revocation (Gate H). Next: **TECH-ID-06 ‚Äî Per-Room Aliases** (not started).
