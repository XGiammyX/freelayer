## Source research

RESEARCH-ID-01 (PR #49) and [DEVICE_KEY_MODEL_IMPLEMENTATION_RESEARCH.md](../research/DEVICE_KEY_MODEL_IMPLEMENTATION_RESEARCH.md) inform this task, verified live 2026-07-23 with versions recorded: **NIST SP 800-63B Rev. 4** authenticator-event discipline (binding requires existing authenticated authority; loss/theft/compromise triggers prompt invalidation; expiration/revocation are explicit end-states — §6.1), **Signal Sesame** (devices are distinct endpoints, add/remove is ongoing state management, not one-time setup — §6.2), **Matrix device keys + cross-signing** (a device key ≠ the user/master key; identifiers ≠ verification; disambiguate by opaque id; a published signature graph is a correlation surface to avoid — §6.3), and **RFC 9420/9750 (MLS)** (a client is a key-holding agent, not a person; membership is logical, credentials are cryptographic; the two layers must not be conflated — §6.4). NIST syncable-authenticator guidance (§6.5) motivates the default of **independent subordinate device keys, never a root copied/synced through a provider fabric**. These are **translated design principles, not conformance claims** — no AAL labels, no compliance assertion; re-verify against the primary source before any external citation.

## The deliverable (record class + slots + labels)

TECH-ID-07 introduces exactly one authoritative local record class in a **separate** aggregate `DeviceKeyModelStateV1` (`persistenceClass: "memory_only" | "null"`; holds authorizations + key slots + labels — [`device-types.ts`](../../packages/identity/src/devices/device-types.ts)), keyed to a caller-resolved `DeviceRootRefV1`. It is not a key, not a signature, not an attestation, not identity, not DevicePosture, not a `RoomMemberRef`, and not a hardware identifier.

`DeviceAuthorizationRecordV1` is a local, non-cryptographic statement that **exactly one** identity root (`rootId` + `rootKind: LocalIdentityRootKindV1`) subordinately authorizes one local device context (`deviceRef`). In production the `deviceClass` is `"current_installation_placeholder"` (the other classes — `linked_device_future` / `temporary_device_future` / `read_only_device_future` / `cold_authorizer_future` — are declared but unreachable). It fixes `cryptographicallyAuthorized: false`, `hardwareAttested: false`, `remotelySynchronized: false`, `origin: "local_root_bootstrap_placeholder"`, and `authorizationProofState: "local_unverified_placeholder"` (the alternative `not_implemented_tech_id_08_gate_f` marks the future passport path). Its lifecycle is `draft_local → active_local_unverified → restricted_local → compromised_suspected → revoked_tombstone` (terminal), enforced by an explicit transition table ([`device-lifecycle.ts`](../../packages/identity/src/devices/device-lifecycle.ts)). Bootstrap admits the first device directly to `active_local_unverified`; re-restriction (`restricted_local → restricted_local`) narrows further; `restricted_local → active_local_unverified` is a **future** reviewed flow and is **not reachable** here.

## Identifiers (opaque, local, non-hardware)

Device references are opaque local ids ([`device-identifiers.ts`](../../packages/identity/src/devices/device-identifiers.ts)): `LocalDeviceRef` (`dev-…`), `DeviceAuthorizationId` (`devauth-…`), `DeviceKeySlotId` (`devslot-…`), each `^[a-z0-9][a-z0-9_-]{2,127}$`, length 4–128, no `@`. They encode **no** hardware/OS data (serial/model/MAC/IMEI/advertising id/hostname), no root/persona/room/member data, no timestamp, no public key/fingerprint, no posture, and no authority; they are never derived from a (future) public key and have no public-discovery meaning. Ids are **injected** into the reducer (`DeviceGeneratedIdsV1`), never minted with randomness inside it.

## Scopes + capabilities (attenuation-only, one root, no wildcard)

A `DeviceAuthorizationScopeV1` ([`device-scopes.ts`](../../packages/identity/src/devices/device-scopes.ts)) is the **maximum** identity context a device may later operate within: kinds `identity_root | persona | relationship | room | read_only`, each referencing **exactly one** `rootId`. `validateDeviceAuthorizationScopeV1` fail-closes the shape (the narrower ids present must exactly match the kind; `read_only` must carry no mutation authority); there is **no wildcard, no multi-root, and no cross-root scope**, and no scope inferred from a label or DevicePosture. Narrowing is **monotonic**: `deviceScopeContainsV1` + `attenuateDeviceAuthorizationScopeV1` return the requested scope only if it is equal-or-narrower under the **same root** — widening (`scope_widening_forbidden`), cross-root (`cross_root_forbidden`), and read-only→writable transitions all reject. Capabilities (`DeviceIdentityCapabilityV1`, [`device-types.ts`](../../packages/identity/src/devices/device-types.ts)) are **local constraints, never proof**, and are likewise attenuation-only (`attenuateDeviceCapabilitiesV1` — remove-only, dedup, known values only — [`device-authorization-context.ts`](../../packages/identity/src/devices/device-authorization-context.ts)). The two highest-authority capabilities `identity.device.add_future` and `identity.recovery.approve_future` are structurally **unavailable** (`DEVICE_UNAVAILABLE_CAPABILITIES`); the validator rejects them at bootstrap.

## Key slots + purposes (typed placeholders, NO material)

A `DeviceKeySlotV1` ([`device-key-slots.ts`](../../packages/identity/src/devices/device-key-slots.ts)) is a typed placeholder for **one** future key purpose — it holds **no** key material: `algorithm: "not_selected_gate_f"`, `publicMaterialPresent: false`, `privateMaterialPresent: false`, `hardwareBacked: "unknown_not_assessed"`, `exportability: "not_applicable_no_material"`, `keyReuseAcrossPurposes: "forbidden"`. Purposes (`DeviceKeyPurposeV1` — [`device-key-purposes.ts`](../../packages/identity/src/devices/device-key-purposes.ts)) are six distinct, immutable roles (`device_authorization_signing_future`, `identity_assertion_signing_future`, `pairwise_messaging_future`, `room_messaging_future`, `local_storage_wrapping_future`, `device_linking_future`); **one purpose per slot**, and the validator rejects a duplicate purpose (`duplicate_purpose`) or an unknown one (`unknown_purpose`). Production creates only `reserved_not_generated` slots; the lifecycle union reserves `active_future` / `rotation_required_future` / `compromised_suspected` / `revoked_tombstone` for Gate F. Real key generation, algorithm selection, and any signature are Gate F.

## Bootstrap (single, non-authorizing, exact-scope)

Bootstrap is the only creation path: `identity.device.bootstrap_current_installation_placeholder` ([`device-commands.ts`](../../packages/identity/src/devices/device-commands.ts)). The reducer ([`device-reducer.ts`](../../packages/identity/src/devices/device-reducer.ts)) requires a usable caller-resolved root ref (`rootActive && !rootCompromised`), matches `command.rootId` and `expectedRootRevision` against it, and **refuses** a second record for a root that already has one (`DeviceAdditionUnavailableError` — `device_already_exists`). It writes one `active_local_unverified`, `cryptographicallyAuthorized: false` placeholder plus one `reserved_not_generated` empty slot per requested purpose (with a slot-id count check). The command is future-gated against everything else: `add_remote` / `link` / `approve_qr` / `import` / `copy_root_secret` reject as `DeviceAdditionUnavailableError`, and `issue_passport` / `verify_attestation` reject as `DevicePassportUnavailableError` ([`device-validation.ts`](../../packages/identity/src/devices/device-validation.ts)). A device cannot promote itself, enroll a peer, or become a cryptographic authority; a first-device bootstrap is subordinate, non-authorizing, and cannot be repeated.

## Lifecycle operations (restrict / mark_compromised / revoke)

Three mutation paths beyond bootstrap, each revision-checked and terminal-aware ([`device-reducer.ts`](../../packages/identity/src/devices/device-reducer.ts)): **restrict** narrows scope + capabilities (attenuation-only) and moves to `restricted_local`; **mark_compromised** moves to `compromised_suspected` (which denies ordinary operations); **revoke** moves to the terminal `revoked_tombstone` and marks the device's key slots `revoked_tombstone`. Each bumps `IdentityLocalRevision` by one. A revoked record cannot reactivate, and `LocalDeviceRevocationResultV1` discloses the honest local-only blast radius: `revokedLocally: true`, `localSnapshotsInvalidated: true`, and **no** remote device / remote session / message-key rotation / room-key rotation / data erase / cryptographic revocation (`remoteDevicesAffected` … `cryptographicRevocationPerformed` all `false`, `redacted: true`). Revocation is a local state transition, not a delete, and makes no remote or forensic-erasure claim.

## Stale-context invalidation (fail-closed snapshot)

A caller captures a `DeviceAuthorizationSnapshotV1` (`cryptographicallyAuthorized: false` — a snapshot is not a proof) to authorize a later operation; `assertDeviceAuthorizationSnapshotCurrentV1` re-checks it against the **authoritative** current record + root ref immediately before execution and fails **closed** on any drift ([`device-authorization-context.ts`](../../packages/identity/src/devices/device-authorization-context.ts)): identity mismatch across snapshot/current/root, a not-usable root (`!rootActive || rootCompromised`), a moved revision, a non-operational lifecycle, DevicePosture/room-policy **tightening** (`postureTightens: true` denies — posture never grants), a removed required capability, or a scope that no longer covers the requested operation. Stale authorization is **never** silently refreshed. This is the local fence — not distributed revocation, not a trusted timestamp, not a cryptographic continuity claim.

## Root vs device compromise (modeled separately)

Root compromise and device compromise are **separate** states: the device lifecycle (`compromised_suspected` / `revoked_tombstone`) is distinct from the root lifecycle, and marking one does not auto-mark the other. But the hierarchy is enforced downward — a not-usable root (`rootActive: false`, e.g. an ephemeral process epoch that passed, or `rootCompromised: true`) invalidates **all** child device operations through both `requireRootRef` (bootstrap) and the snapshot check. A device event never revokes the root; a root revocation/compromise invalidates its devices. The correct blast radius of a *real* key compromise is a Gate F property, not decided here.

## Command validation (fail-closed)

`validateDeviceKeyModelCommandV1` ([`device-validation.ts`](../../packages/identity/src/devices/device-validation.ts)) is a closed discriminated union of **6** commands (bootstrap / restrict / mark_compromised / revoke / label.set / label.clear — [`device-commands.ts`](../../packages/identity/src/devices/device-commands.ts)); there is **no** generic device patch. It enforces exact schema version, exact discriminator, and exact keys (`assertExactKeys`), and rejects dangerous keys (`__proto__`/`prototype`/`constructor`) plus four purpose-built forbidden sets with specific errors: **key material** (`publicKey`/`privateKey`/`secretKey`/`seed`/`mnemonic`/`signature`/`certificate` → `DeviceKeyMaterialForbiddenError`), **hardware/OS identifiers** (`hardwareId`/`serialNumber`/`imei`/`macAddress`/`advertisingId`/`hostname`/`osBuild`/`attestation` → `HardwareIdentifierForbiddenError`), **posture** (`devicePosture`/`screenShieldActive` → `DevicePostureAuthorityError`), and **authority booleans** (`trusted`/`verified`/`primary`/`owner`/`admin`/`superuser`/`linked`/`passport`/`remote`/`synchronized` → validation reject). The caller cannot set proof/trust/key/synchronization state directly; scope, capability, and key-purpose shapes are validated exhaustively; every mutation carries the expected root and/or device-authorization revision; and label text is normalized through the reused TECH-ID-05 normalizer with raw input never appearing in errors. It is explicitly **not** a hostile-input parser (Gate E).

## Policy + PolicyDecision integration (exact-scope)

`resolveDeviceKeyModelPolicyV1` ([`device-policy.ts`](../../packages/identity/src/devices/device-policy.ts)) is pure, deterministic, fail-closed, strictest-wins across the seven privacy modes over eleven operations (bootstrap / restrict / mark_compromised / revoke / label.set / label.clear / summary.list / summary.detail / key_slots.read_redacted, plus the future-gated `add_device_future` / `issue_passport_future`). Everything is `memory_only | null`. **Bunker** and **Emergency** deny expansive bootstrap (`denyExpansive`); **Emergency** permits only restrictive ops (restrict / mark_compromised / revoke / label.clear / the redacted reads) and forces `maximumActiveDevicePlaceholders: 0`; local **labels** deny in Private / Ghost / Bunker / Emergency; strict modes suppress exact counts/timestamps and drive summary redaction. The structural fields are `false` in **every** mode: `realKeysAvailable`, `deviceAdditionAvailable`, `devicePassportAvailable`, `remoteRevocationAvailable`, `hardwareAttestationAvailable`, `persistentPlaintextAllowed`, `networkAllowed`, `notificationAllowed`, `telemetryAllowed`, `aiAllowed`. The pipeline `applyDeviceKeyModelCommandV1` ([`device-pipeline.ts`](../../packages/identity/src/devices/device-pipeline.ts)) maps each command onto an exact `identity.device.*` side-effect scope (`SCOPE_FOR_COMMAND`: bootstrap→`identity.device.bootstrap`, restrict→`identity.device.restrict`, mark_compromised→`identity.device.mark_compromised`, revoke→`identity.device.revoke`, label.set→`identity.device.label.write`, label.clear→`identity.device.label.clear`) and requires an authentic, allowed, **exact-scope** `PolicyDecision` — a forged, wrong-scope, or denied decision is rejected **before** mutation. Reads require their own decisions (`identity.device.summary.read` — [`device-summary.ts`](../../packages/identity/src/devices/device-summary.ts)). In total the privacy package defines **8** `identity.device.*` `PolicySideEffectScope` values ([`packages/privacy/src/index.ts`](../../packages/privacy/src/index.ts): `bootstrap`, `restrict`, `mark_compromised`, `revoke`, `label.write`, `label.clear`, `summary.read`, `key_slots.read_redacted`). A device id, label, `RoomMemberRef`, room role, or DevicePosture **never** authorizes; posture and role may only **tighten**.

## Pure reducer + memory/null repositories

`reduceDeviceKeyModelCommandV1` ([`device-reducer.ts`](../../packages/identity/src/devices/device-reducer.ts)) is pure and deterministic: validate-before-mutate; injected ids (`DeviceGeneratedIdsV1`) + injected clock + a validated `DeviceRootRefV1` only; no storage/network/notification/AI/crypto/randomness; **no** key material ever; **no** RoomOS/Secure-Device calls; source state/command never mutated; revisions increment by one; failure yields no partial state. `InMemoryDeviceKeyModelRepositoryV1` and `NullDeviceKeyModelRepositoryV1` ([`device-repository.ts`](../../packages/identity/src/devices/device-repository.ts)) are memory-only / null — no disk, browser storage, database, hidden key store, serialized device list, persistent label history, or hardware-identifier collection — with deep defensive `structuredClone`, no global singleton. Every repository operation re-checks `isAuthentic`: an authentic decision, `verdict === "allowed"`, and a `sideEffect` beginning with `identity.device.` — a non-device scope (including another identity domain's `identity.room_alias.*` or a `storage.write` scope) is refused (`DeviceAuthorizationDecisionMismatchError`).

## Labels, summaries, redaction, audit

A `LocalDeviceLabelV1` ([`device-types.ts`](../../packages/identity/src/devices/device-types.ts)) is a **private** recognition note — `visibility: "local_only"`, `peerShared: false`, `authority: "none"`, `postureEvidence: false`; its text is normalized through the reused TECH-ID-05 normalizer and never shared, logged, or treated as a model/OS identity. Summaries are safe by construction ([`device-summary.ts`](../../packages/identity/src/devices/device-summary.ts)): `buildLocalDeviceSummaryV1` / `listLocalDeviceSummariesV1` fix the trust flags `cryptographicallyAuthorized` / `hardwareAttested` / `postureVerified` / `remotelySynchronized` to `false`, expose coarse lifecycle / class / scope-kind, and in strict modes omit the device ref, label, scope details, and counts (`redacted: true`); a read needs its own `identity.device.summary.read` decision (`assertDeviceSummaryDecisionV1`). `readRedactedKeySlotsV1` returns purpose + lifecycle only — never a slot id, algorithm, or material. `DeviceKeyModelError` ([`device-errors.ts`](../../packages/identity/src/devices/device-errors.ts)) carries only a stable code from a 15-member union plus the single static `SAFE_DEVICE_REJECTION` string — never a device/root/persona/room id where avoidable, a label, hardware/model/OS data, key-slot contents, a command/state dump, or a leak sentinel. Audit events are `redacted: true` (category + outcome + reason code only).

## Secure Device separation

Preserved. The devices module imports only `@freelayer/privacy` (+ transitively `@freelayer/security` via identifiers), sibling identity identifiers/types, and the reused TECH-ID-05 normalizer; it imports **no** Secure Device / DevicePosture / ScreenShield / attestation implementation, no UI, no transport, no crypto, no platform API, and — critically — **no `@freelayer/rooms`.** The model reads **no** posture and is never a posture input; a DeviceKey record is a *root-authorization* fact, DevicePosture is an *endpoint-environment* fact, and the two never conflate. The policy's honest `hardwareAttestationAvailable: false` is a disclosure of **absence**, allowlisted by the two existing separation guards. Endpoint protection (key custody, exfiltration resistance, attestation, screenshots) is the **external** Secure Device / Endpoint Defense project's concern; core keeps only contracts, policy inputs, and honest disclosures.

## Known limitations / deferred gates

Stated plainly. A DeviceKey is **not** DevicePosture; DevicePosture is **not** identity; a device is **not** a person; a device reference is **not** a hardware identifier; a device label is **not** authority; a local authorization record is **not** a cryptographic proof; RoomOS membership is **not** device authorization; device revocation is **not** remote erasure. There is **no real key security** (no key bytes, no signatures, no fingerprints, no attestation, no cross-signing, no device passport — `algorithm: "not_selected_gate_f"`), **no cryptographic authorization** (`cryptographicallyAuthorized: false`; subordination is asserted locally, not proven), **no remote revocation** (a revoke is a local terminal tombstone; no remote sessions/keys/data are affected — reconciliation is Gate H), **no hardware binding** (opaque non-hardware refs; `hardwareId`/`deviceFingerprint`/`attestation` rejected), and **no endpoint guarantee** (custody/exfiltration/attestation are external or absent; memory/null retention does not stop a compromised OS, malware, backups, or crash dumps). This is **not** a safe home for real secrets. Deferred: real keys / signatures / derivation / verification (**Gate F**), the signed root↔device Device Passport and device linking / QR / one-time tokens (**TECH-ID-08** + Gate E/F), the full Identity Firewall facade wiring RoomOS↔device results and Crypto↔slot descriptors (**Gate G**), and synchronization / distributed device revocation (**Gate H**). Verified devices, real key material, signatures, attestation, and device passports are **not** implemented in this task — they are named here only to negate them.

## Verdict

**Complete.** The local, root-authorization-scoped, non-cryptographic device key model is implemented as a separate memory/null aggregate (`DeviceKeyModelStateV1`) keyed to a caller-resolved `DeviceRootRefV1`, holding one `DeviceAuthorizationRecordV1` per bootstrapped current-installation placeholder (referencing exactly one root, `cryptographicallyAuthorized: false`), purpose-separated empty key slots (`not_selected_gate_f`, one purpose each, reuse forbidden), and private local labels (`authority: "none"`). It carries attenuation-only scopes and capabilities, an explicit terminal lifecycle (restrict / mark_compromised / revoke) whose revocation is local-only with no remote or cryptographic effect, a fail-closed stale-context snapshot that denies on any drift and on posture tightening, exact-scope `identity.device.*` gating (8 scopes) enforced at the pipeline and again at the repository, a fail-closed strictest-wins policy with structural absence of keys/addition/passport/remote-revocation/attestation/persistence/network/notification/telemetry/AI, +14 Policy Matrix rows (5 structural denies/future-gates; the device_key/device_passport rows reused as Gate F / TECH-ID-08), a dedicated static guardrail, and honest documentation. A DeviceKey is not DevicePosture, a device is not a person, a device reference is not a hardware identifier, a device label is not authority, a local authorization record is not a cryptographic proof, RoomOS membership is not device authorization, and device revocation is not remote erasure. Enforcement note (see [boundary audit](DEVICE_KEY_MODEL_BOUNDARY_AUDIT.md)): the sdk/core Identity Firewall facade wiring RoomOS↔device results and Crypto↔slot descriptors is a later task (Gate G). Deferred: crypto / real keys / signatures / verification (Gate F), Device Passport + device linking (TECH-ID-08 + Gate E/F), full facade (Gate G), synchronization / distributed revocation (Gate H).
