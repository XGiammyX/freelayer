## The record class (the deliverable)

TECH-ID-06 introduces exactly one local, room-binding-scoped presentation record in a **separate** aggregate `RoomAliasStateV1` (`persistenceClass: "memory_only" | "null"`) that references a `RoomIdentityBindingV1` **by id** — the binding record is never rewritten ([`room-alias-types.ts`](../../packages/identity/src/room-aliases/room-alias-types.ts)). It is not identity, membership, a role, verification, authentication, a `RoomMemberRef`, a public username, or a cryptographic identifier.

`RoomPresentationAliasV1` is a room-binding-scoped, root/persona-bound, local-only display name for how the local user presents inside **one** room. It carries `roomId: RoomLocalIdRef`, `bindingId`, `rootId`, `personaId`, `rootKind`, optional `membershipId?: RoomMembershipIdRef`, and fixes `scope: "one_room_only"`, `sharingState: "not_shared_tech_id_06"`, `authenticatedBinding: "not_implemented_gate_f"`, and `remoteUpdateState: "not_implemented_gate_e_f_h"`. Its `origin` is `"user_supplied_local" | "injected_generated_placeholder"` — never an automatic copy of a contact alias, persona label, another room's alias, the root id, or a `RoomMemberRef`. Lifecycle `draft_local → active_local_unshared → retired_tombstone` (terminal), enforced by an explicit transition table ([`room-alias-lifecycle.ts`](../../packages/identity/src/room-aliases/room-alias-lifecycle.ts)). At most **one** active (non-retired) alias per binding is enforced in the reducer (`hasActiveAliasForBinding`, plus a second `duplicate_active` guard on activate). It is never transmitted, negotiated, or authenticated to the room or its members in this task.

RoomOS role/state values are **mirrored** locally as a narrow caller-supplied read model (`RoomOsMembershipRoleV1` / `RoomOsMembershipStateV1`); identity never imports `@freelayer/rooms`. A secondary private per-member note (the room analogue of TECH-ID-05's local peer label) is deliberately **out of scope** — RoomOS holds no member display name, and labeling another member is a future consumer concern.

## Text normalization (reused, conservative, honest)

The room alias **reuses `normalizeContactAliasDisplayTextV1` and `ContactAliasTextV1` verbatim** ([`room-alias-validation.ts`](../../packages/identity/src/room-aliases/room-alias-validation.ts), [`room-alias-reducer.ts`](../../packages/identity/src/room-aliases/room-alias-reducer.ts)); no new normalizer was written. It applies **one** transform — `input.normalize("NFC").trim()` (NFC, **not** NFKC) — then rejects dangerous code points: C0 controls `U+0000..U+001F` (incl. **NUL**), DEL + C1 `U+007F..U+009F`, line/paragraph separators `U+2028`/`U+2029`, bidi overrides `U+202A..U+202E`, bidi isolates `U+2066..U+2069`, zero-width space `U+200B`, word joiner `U+2060`, and BOM/ZWNBSP `U+FEFF`; **ZWJ `U+200D` and ZWNJ `U+200C` are retained** (Persian, Devanagari, emoji ZWJ sequences). Bounds 1..64 scalars, ≤256 UTF-8 bytes; raw input is discarded. It does **not** case-fold, transliterate, or block whole scripts, and is **not** a UTS #39 confusable detector — baseline Unicode validation is **not** full spoofing prevention, and a mixed-script or look-alike name can still be accepted.

## Lifecycle & rotation

Transition tables are explicit and terminal-is-terminal; unknown/self transitions throw `RoomAliasLifecycleError` ([`room-alias-lifecycle.ts`](../../packages/identity/src/room-aliases/room-alias-lifecycle.ts)). **Rotation** (`identity.room_alias.rotate`) is one pure reducer step that retires the old alias and creates a **new** `active_local_unshared` record under the same binding id, revision-checked (`revision_mismatch`) and ref-matched. It preserves continuity — `RoomAliasRotationResultV1 { rotated: true, roomBindingPreserved: true, membershipPreserved: true, membershipRoleChanged: false, assuranceChanged: false, remoteCopiesAffected: false, authenticatedRemoteUpdatePerformed: false, historicalMessagesRelabelled: false, redacted: true }`. Membership, role, and assurance live on RoomOS + the binding, not the alias; rotation cannot touch them, delete remote copies, perform an authenticated update, or relabel prior messages. A retired alias cannot reactivate; a new presentation is a new record.

## Command validation (fail-closed)

`validateRoomAliasCommandV1` ([`room-alias-validation.ts`](../../packages/identity/src/room-aliases/room-alias-validation.ts)) is a closed discriminated union of **4** commands (create / activate / rotate / retire — [`room-alias-commands.ts`](../../packages/identity/src/room-aliases/room-alias-commands.ts)); there is **no** generic room-alias/room-profile patch. It enforces exact schema version, exact discriminator, and exact keys; rejects dangerous keys (`__proto__`/`prototype`/`constructor`) and a `FORBIDDEN_FIELDS` set covering authority/role/membership/public-directory/secret/copy-from caller fields (`verified`, `trusted`, `authenticated`, `owner`, `moderator`, `admin`, `role`, `permissions`, `capabilities`, `global`, `public`, `searchable`, `username`, `phone`, `email`, `did`, `rootPublicId`, `publicKey`, `privateKey`, `contactAliasId`, `copyFromContact`, `copyFromRoom`, `devicePosture`, `screenShieldActive`, `peerShared`). Visible text is normalized through the shared TECH-ID-05 normalizer (raw input never appears in errors); the caller may not set collision/disambiguation/assurance/sharing/authentication state; every non-create update carries the expected current revision, and every binding-scoped write carries the `expectedBindingRevision`.

## Policy + PolicyDecision integration (exact-scope)

`resolveRoomAliasPolicyV1` ([`room-alias-policy.ts`](../../packages/identity/src/room-aliases/room-alias-policy.ts)) is pure, deterministic, fail-closed, strictest-wins across the seven privacy modes over **7** operations (create / activate / rotate / retire, `display_context.read`, `collision_assessment.read`, `reuse_assessment.read`). Bunker/Emergency deny expansive writes (create/activate/rotate) and the collision/reuse assessments; Emergency permits only restrictive ops (`retire`, `display_context.read`); strict modes suppress exact timestamps and drive display redaction. `networkSharingAllowed`, `authenticatedBindingAvailable`, `publicDirectoryAllowed`, `globalUsernameAllowed`, `persistentPlaintextAllowed`, `notificationAllowed`, `telemetryAllowed`, and `aiAllowed` are **structurally `false` in every mode**. The pipeline `applyRoomAliasCommandV1` ([`room-alias-pipeline.ts`](../../packages/identity/src/room-aliases/room-alias-pipeline.ts)) maps each write onto an exact `identity.room_alias.*` side-effect scope (`create` → `create`; `activate` and `retire` → the shared `lifecycle` scope; `rotate` → `rotate`) and requires an authentic, allowed, **exact-scope** `PolicyDecision` — a forged, wrong-scope, or denied decision is rejected **before** mutation. The display read requires its own `identity.room_alias.display_context.read` decision ([`room-alias-summary.ts`](../../packages/identity/src/room-aliases/room-alias-summary.ts)). In total the privacy package defines **6** `identity.room_alias.*` `PolicySideEffectScope` values ([`packages/privacy/src/index.ts`](../../packages/privacy/src/index.ts): `create`, `lifecycle`, `rotate`, `display_context.read`, `collision_assessment.read`, `reuse_assessment.read`). An alias id, display text, binding id, room role, `RoomMemberRef`, or DevicePosture never authorizes; an alias-text collision never grants authority; room policy and DevicePosture may only **tighten**, never grant.

## Pure reducer + caller-resolved binding ref + memory/null repositories

`reduceRoomAliasCommandV1` ([`room-alias-reducer.ts`](../../packages/identity/src/room-aliases/room-alias-reducer.ts)) is pure and deterministic: validate-before-mutate; injected ids (`RoomAliasGeneratedIdsV1`) + injected clock + a validated `RoomAliasBindingRefV1` only; no storage/network/notification/AI/crypto/randomness; **no direct RoomOS membership mutation**; source state/command never mutated; revisions increment by one; failure yields no partial state. The `RoomAliasBindingRefV1` is the discipline that keeps aliases honest: it carries `rootKind` + `bindingRevision` + `rootBindingActive`, and `requireRef` rejects an inactive/tombstoned binding or a removed membership — so an **ephemeral alias can never outlive its root** (the caller sets `rootBindingActive: false` once the process epoch passes). Ids are opaque, injected, and non-derived (`ralias-…`; `^[a-z0-9][a-z0-9_-]{2,127}$`, no `@`; encode no display text, root/persona/binding/membership id, room id, phone/email/username, role, timestamp, device info, or key material — [`room-alias-identifiers.ts`](../../packages/identity/src/room-aliases/room-alias-identifiers.ts)). `InMemoryRoomAliasRepositoryV1` and `NullRoomAliasRepositoryV1` ([`room-alias-repository.ts`](../../packages/identity/src/room-aliases/room-alias-repository.ts)) are memory-only / null — no disk, browser storage, database, encrypted storage, collision/reuse index, or alias history — with deep defensive `structuredClone`, no global singleton, no serialized snapshot. Every repository operation re-checks `isAuthentic`: an authentic decision, `verdict === "allowed"`, and a `sideEffect` beginning with `identity.room_alias.` — a non-room-alias scope (including a TECH-ID-05 `identity.alias.*` scope) is refused (`decision_mismatch`).

## Collision, disambiguation, and cross-room reuse

`assessRoomAliasCollisionV1` and `assessRoomAliasReuseV1` ([`room-alias-collision.ts`](../../packages/identity/src/room-aliases/room-alias-collision.ts)) are **privacy diagnostics only**. Room-local collision compares the target normalized value against other **current** members' aliases in a single-call transient set (fail-closed exclusion of `removed_tombstone` members, missing aliases, and the target member itself); a duplicate is **allowed** and returns `exactCollisionCountExposed: false` / `collidingMemberIdsExposed: false` / `impersonationProven: false`. Cross-room reuse reports whether the same normalized value is an active alias in a **different** room, returning `affectedRoomIdsExposed: false` / `affectedRoomCountExposed: false` / `unlinkabilityGuaranteed: false`; strict modes skip evaluation (`not_evaluated`). `resolveRoomAliasDisambiguationV1` ([`room-alias-disambiguation.ts`](../../packages/identity/src/room-aliases/room-alias-disambiguation.ts)) forbids `alias_only` on a duplicate, never exposes a global/root/raw-membership identifier, and falls back to a redaction **requirement** rather than inventing an identifier when no safe room-local reference exists. A duplicate is **not** proof of impersonation; a unique value is **not** proof of authenticity or unlinkability; disambiguation is **not** verification.

## Binding integration + honest cross-package consistency limitation

The binding schema migrated `RoomIdentityBindingV1.roomAliasState` (and the ephemeral `EphemeralRoomIdentityBindingV1.roomAliasState`) from the `"not_implemented_tech_id_06"` placeholder to `RoomAliasBindingStateV1 = "none" | "local_alias_v1" | "retired"` (default `"none"`; local, memory-only, unreleased — no persisted data to migrate) ([`identity-types.ts`](../../packages/identity/src/identity-types.ts); [`ephemeral-types.ts`](../../packages/identity/src/ephemeral/ephemeral-types.ts), [`ephemeral-reducer.ts`](../../packages/identity/src/ephemeral/ephemeral-reducer.ts)). The alias **record** (`RoomAliasStateV1`) is authoritative; `roomAliasState` is a **derived hint** computed by `deriveRoomAliasBindingStateV1` ([`room-alias-summary.ts`](../../packages/identity/src/room-aliases/room-alias-summary.ts)), never a second source of truth. Stated honestly: there is **no true cross-package atomicity** between the Identity Firewall alias record and RoomOS membership. The module reads a caller-supplied RoomOS snapshot rather than importing `@freelayer/rooms`, so a membership can change between snapshot resolution and commit; RoomOS remains authoritative for membership/role, the Identity Firewall for the alias record, and the pipeline's documented ordering narrows but cannot close that window. Genuine binding authenticity and synchronized revocation are Gate F/H.

## Redacted errors / audit / display context

`RoomAliasError` ([`room-alias-errors.ts`](../../packages/identity/src/room-aliases/room-alias-errors.ts)) carries only a stable code from a 13-member union (`validation`, `policy_denied`, `decision_mismatch`, `not_found`, `revision_mismatch`, `lifecycle`, `binding_mismatch`, `membership_mismatch`, `duplicate_active`, `transfer_forbidden`, `collision`, `remote_sharing_unavailable`, `authentication_unavailable`) plus the single static `SAFE_ROOM_ALIAS_REJECTION` string — never alias text, a binding/root/persona/member/room id where avoidable, a command payload, collision details, or a leak sentinel. `buildRoomMemberDisplayContextV1` ([`room-alias-summary.ts`](../../packages/identity/src/room-aliases/room-alias-summary.ts)) returns `membershipRole` / `membershipState` / `identityAssurance` **separately** from alias text, never a generic "verified member" flag, fixes `aliasVerified: false` / `realWorldIdentityVerified: false` / `cryptographicBindingAvailable: false` / `authorityDerivedFromAlias: false`, forces a safe disambiguation strategy on a duplicate, and omits the room id + alias text in strict modes. Audit events are `redacted: true` (category + outcome + reason code only). Security-sensitive callers are told explicitly never to use a room alias as the sole identity signal for role changes, membership removal, verification, recovery, device linking, export, or destructive actions.

## Secure Device separation

Preserved. The room-alias module imports only `@freelayer/privacy` (+ transitively `@freelayer/security` via identifiers), sibling identity types, and the reused TECH-ID-05 normalization; it imports **no** Secure Device / DevicePosture / ScreenShield / attestation implementation, no UI, no transport, no crypto, and — critically — **no `@freelayer/rooms`.** DevicePosture remains tighten-only in policy and a rejected caller field in validation; a device posture or room role can restrict alias operations but can never validate, sign, or bless alias text. Endpoint protection (screenshots, task-switcher previews, secure input, memory hardening) is the **external** Secure Device / Endpoint Defense project's concern; the core keeps only contracts/policy inputs/honest disclosures.

## Known limitations / deferred gates

Stated plainly. A room alias is **not** identity, **not** membership, **not** a role, **not** verification, **not** authentication, **not** a `RoomMemberRef`, **not** a public/global username, and **not** a cryptographic identifier; it does **not** prove a person controls a key and does **not** grant admission. **A room role is not proof of identity; a name collision is not proof of impersonation; disambiguation is not verification; DevicePosture is not identity.** Normalization is conservative baseline hygiene — baseline Unicode validation is **not** full spoofing prevention, it is **not** a UTS #39 confusable/homograph detector, and a mixed-script or look-alike name can still be accepted; security-sensitive UI must never rely on alias text alone. The cross-room reuse warning detects only identical *normalized* text in the current local vault, exposes no room graph, and a visible room alias is **not** cryptographic unlinkability. There is **no** remote room-alias deletion, sharing, or synchronization, and **no** historical-message relabelling; rotation changes neither old messages nor peer copies. There is **no true cross-package atomicity** with RoomOS membership; RoomOS stays authoritative for membership/role. Room aliases are **memory/null only and not safe for real secrets**; endpoint exposure (screenshots, second devices, crash dumps, swap) is external or inherent. Deferred: cryptographic alias authenticity / key material (**Gate F**), wire formats / remote room-alias exchange (**Gate E**), the full Identity Firewall facade wiring RoomOS↔alias results and device authorization (**Gate G**), and synchronization / distributed revocation / historical relabelling (**Gate H**). Verified identity, device passports, recovery, identity keys, key transparency, and DID are **not** implemented in this task — they are named here only to negate them.
